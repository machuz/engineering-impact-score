EIS と DORA・SPACE・LinearB —— 観測する対象の違い

「結局、その指標は何を見ようとしているのか」で整理する。

観測対象がそもそも違う

エンジニアリングの数値化を掲げるフレームワークは多い。ただ、何を観測しているかで切ると、実は別物の道具が並んでいるだけのことが多い。

道具 観測対象 データ源 立場
DORAチームの出力速度CI/CD パイプライン管理
SPACEチームの状態（5 次元）サーベイ＋定量混成評価
LinearBチームの生産性DORA + SPACE を統合管理・評価
EIS個人の痕跡の重力git log / git blame のみ観測

DORA が答えない問い

DORA は優れた道具だ。だが答えられない問いがひとつある —— 「このチームが強いのは、誰がいるからか？」 DORA はチームという単位でしか見ない。チームの速度が出ているとき、その速度を作っているのは誰か、という問いには手が届かない。

SPACE はそこに踏み込もうとしたが、主観サーベイを混ぜた結果、自己申告と上司の印象に依存した計測に戻った。個人に迫ったが、自己申告の壁で止まった。

LinearB は DORA + SPACE の合成で、すべての生産性指標を一画面にまとめる。強力だが、根っこの問いは変わらない —— チーム単位の出力を、見やすく見せているだけ。

EIS は何を観測するか

EIS は個人の痕跡だけを観測する。具体的には：

  • 生存したコード（時間減衰 blame）
  • 設計ファイルへの貢献（アーキテクチャの骨格）
  • 他者の負債の清掃率（見えにくい貢献）
  • モジュール所有率（バスファクター）
  • ほか 3 軸で、計 7 軸

データ源は git log と git blame のみ。外部 API も AI トークンも不要。自己申告も入らない。

決定的な違い —— 評価を混ぜない

DORA・SPACE・LinearB はいずれも管理のための道具だ。数字を見て、どのチームに介入するかを決める。評価の回路に直結している。

EIS は意図的に観測と評価を分離する。出てきたシグナルを見て何を判断するかは、後工程のレイヤー（SaaS や組織）に委ねる。CLI 自体は望遠鏡のままで、裁判所にはならない。

CLI は望遠鏡、SaaS は天文台。観測であって、評価ではない。

もうひとつの違い —— ゲーム耐性

EIS の Survival 軸は時間減衰する。急いで書き捨てたコードは減衰で消え、残ったコードだけがカウントされる。コミット数や行数を稼いでも、後で消される仕事はスコアに残らない。

指標を追いかけるとゲームになる、という批判はこの手の計器に常に付きまとうが、EIS は望遠鏡を曲げないための設計を優先した。
