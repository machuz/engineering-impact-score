# git考古学 #15 —— The Age of AI：AIがもたらす星の爆発

![Cover](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/blog/hatena/cover-ch15.png?v=1)

*AIは星を作る。しかし重力は作らない。*

### 前章までのあらすじ

[第14章](https://ma2k8.hateblo.jp/entry/2026/03/15/062612)ではCivilization——なぜ一部のコードベースだけが文明になるのか——について書いた。

文明の三要素：Architect（構造を作る）、Anchor（秩序を維持する）、Producer（領土を拡張する）。

そして今、この方程式に新しい変数が加わろうとしている。

---

## 星の爆発

ソフトウェア宇宙は今、大きな変化の時代に入っている。

**AI**である。

AIはコードを書く。しかも人間より速く、人間より大量に。GitHub Copilot、Claude Code、Cursor——AIアシスタントはもはや「実験的な補助ツール」ではない。プロダクションコードのかなりの割合がAIの手で生成される時代が、もう来ている。

宇宙の言葉で言えば、これは**星の爆発**だ。

新しいコード。新しい関数。新しいサービス。新しいファイル。コード宇宙には、これまでの歴史の中で経験したことのない速度で、大量の星が生まれている。

---

## しかし、星の数だけでは宇宙は成立しない

第7章で書いたように、コード宇宙は星（コード）だけでは構成されない。重力（構造的影響力）があり、ダークマター（見えない力）があり、エントロピー（無秩序への傾向）がある。

星が増えるほど、**エントロピーも増える**。

これは直感に反するかもしれない。コードが増えるのは良いことではないのか？　生産性が上がるのは良いことではないのか？

答えはNoだ。少なくとも、無条件にはNoだ。

### AIが生むコードの特性

AIが生成するコードには、構造的に重要な特性がある。

**AIは未来を生成するが、過去に責任を持たない。**

人間のエンジニアがコードを書くとき、そこには暗黙の文脈がある。「このモジュールはこういう設計思想で作られている」「この命名規則にはこういう意図がある」「ここをこう書くのは、3ヶ月前のあの議論の結果だ」——これらの文脈は、git logには残らない。エンジニアの頭の中にある。

AIにはこの文脈がない。AIはプロンプトに書かれた要件を満たすコードを生成する。そのコードは動く。テストも通る。しかし、**既存の構造との整合性は保証されない**。

EISの言葉で言えば：

```
AI generates stars    ← Production
AI does not generate gravity    ← Design, Survival
```

AIはProduction軸を爆発させる。しかしDesign軸とSurvival軸には寄与しない——むしろ、構造に合わないコードが混入することで、これらの軸を**毀損する**可能性がある。

---

## AIコードのSurvival問題

第1章で、EISの最重要軸はSurvival（コード生存力）だと書いた。書いたコードが時間を超えて残り続けるかどうか。

AIが書いたコードのSurvivalはどうなるか。

これはまだ十分なデータがないが、仮説は立てられる。

### 短期的にはSurvivalが高い

AIが生成したコードは、多くの場合「動く」。テストも書ける。だから短期的には書き直される理由がない。Survivalは高く出る。

### 長期的にはSurvivalが下がる

しかし数ヶ月後、隣接するモジュールに変更が入ったとき何が起きるか。AIが書いたコードは、設計の意図を理解して書かれていない。だから**構造的な変更に耐えられない**。結果、書き直される。Survivalが下がる。

つまりAIコードのSurvivalは、**時間軸で劣化するパターン**を示す可能性がある。短期的には生き残るが、構造的な変化の波が来ると崩れる。

EISのタイムラインで見ると、こう見えるかもしれない：

```
期間        Survival
2025-Q2     85    ← AI生成直後、書き直されていない
2025-Q3     72    ← 隣接モジュール変更で一部書き直し
2025-Q4     55    ← 設計変更に耐えられず大幅リライト
2026-Q1     40    ← 「もう一から書き直した方が早い」
```

これは第11章のエントロピーそのものだ。**構造なきコードは、構造の変化に対して脆い。**

## AIはArchitectになれるか

ここが核心だ。

前章で書いた文明の方程式を思い出してほしい。

```
Civilization =
  Architect  → creates gravity (structure)
  + Anchor   → maintains order (stability)
  + Producer → expands territory (growth)
```

AIは明らかにProducerの役割を果たせる。コードを書く。領土を拡げる。速く、大量に。

AIはAnchorの役割を**部分的に**果たせるかもしれない。リファクタリング、テスト追加、lint修正——秩序の維持作業の一部はAIに任せられる。

しかし、**AIはArchitectになれるか？**

現時点での答えは**No**だと私は考えている。

Architectは構造を**定義する**。モジュール境界をどこに引くか。依存の方向をどうするか。命名規則に何を込めるか。これらの判断には、ドメイン知識、チームの文脈、ビジネスの方向性、技術的制約——複数の次元の知識が必要だ。

AIは「このコードをどう書くか」には答えられる。しかし「このシステムをどう構造化するか」には、まだ答えられない。

**コードを書くことと、重力を生むことは、全く別の能力だ。**

---

## エンジニアの価値の転換

これが、AI時代にエンジニアの価値がどう変わるかの答えだ。

### AI以前

エンジニアの価値 = **コードを書く能力**

速くコードを書ける人が強い。大量に書ける人が強い。Production軸が高い人が評価される。

### AI以後

エンジニアの価値 = **重力を生む能力**

AIがコードを書く時代に、人間がコードを書くことの価値は相対的に下がる。しかし、構造を設計する能力——重力を生む能力——の価値は下がらない。むしろ**上がる**。

なぜなら、AIが大量のコードを生成するほど、そのコードに秩序を与える人間の重要性が増すからだ。星が増えるほど、重力が必要になる。

EISの7軸で見ると、AI時代に価値が上がる軸と下がる軸はこうなる：

| 軸 | AI時代の価値変化 | 理由 |
|---|---|---|
| Production | **↓ 低下** | AIが代替可能。人間の生産速度は競争優位にならない |
| Quality | → 維持 | AIのコード品質を検証する力は依然必要 |
| Survival | **↑ 上昇** | 構造に根ざしたコードだけが生き残る。設計力の証 |
| Design | **↑↑ 大幅上昇** | AI時代に最も希少な能力。重力を生む唯一の軸 |
| Breadth | → 維持 | 横断的知識は人間の文脈理解に依存 |
| Debt Cleanup | **↑ 上昇** | AIが生む負債を清掃する能力 |
| Indispensability | → 維持 | 構造的影響力は人間が担い続ける |

**Design軸が最重要軸になる。** 第1章ではSurvivalを最重要軸と書いた。それは今でも正しい。しかしAI時代においては、SurvivalとDesignは表裏一体になる。構造的に正しいコードだけがSurvivalを維持できるからだ。

## 生き残るエンジニアの定義

では、AI時代に「生き残るエンジニア」とは具体的に何なのか。言語化しておきたい。

### コードを書くエンジニアは生き残れない

誤解を恐れずに言う。**「コードを書くことが主な価値」のエンジニアは、AI時代に生き残れない。**

これは能力の否定ではない。市場の構造変化だ。AIが1時間で書けるコードを、人間が1日かけて書くことに経済的な価値がなくなる。速く正確にコードを書く能力は、AI以前は希少資源だった。AI以後は、コモディティになる。

### 生き残るエンジニアの3条件

EISの7軸とソフトウェア宇宙論を通じて見えてきた「生き残るエンジニア」の条件は、3つに集約される。

**1. 重力を生めること（Design）**

構造を定義できる能力。モジュール境界を引き、依存の方向を決め、命名規則にドメインの意味を込められること。第7章の言葉で言えば、**コード宇宙に重力場を作れること**。

AIが大量のコードを生成する時代に、そのコードがバラバラに飛散しないよう、引力を与えられる人間。これが最も希少な能力になる。

**2. 構造の「なぜ」を説明できること（Context）**

コードの「何」はAIが書ける。しかし「なぜこの構造なのか」——ドメインの文脈、ビジネスの制約、チームの歴史的経緯——これを理解し、伝えられるのは人間だけだ。

第10章で書いたダークマター——コミットログに現れない見えない力——は、人間の頭の中にしかない。AIにコンテキストを与え、正しい方向にコードを生成させるには、そのダークマターを言語化できなければならない。

**3. エントロピーと戦えること（Survival × Debt Cleanup）**

AIが生成するコードは、エントロピーを加速させる。構造に合わないコードが混入し、技術的負債が蓄積する。

生き残るエンジニアは、このエントロピーに抗える。AIが生んだコードを構造に沿って再構築し、負債を返済し、秩序を維持できる。第11章のエントロピーの法則に真っ向から対抗できる力。

### エンジニアの新しい分類

これを整理すると、AI時代のエンジニアは3つに分類される。

| 分類 | 特性 | AI時代の展望 |
|---|---|---|
| **Code Writer** | コードを書くことが主な価値 | AIに代替される。市場価値が急落 |
| **Code Architect** | 構造を設計し、重力を生む | AI時代に最も希少。市場価値が急騰 |
| **Code Custodian** | 構造を維持し、エントロピーと戦う | AIが生む負債の清掃者。価値が上昇 |

第14章の文明の方程式に対応させると：

- Code Writer = Producer
- Code Architect = Architect
- Code Custodian = Anchor

**AI時代に生き残るのは、ArchitectとAnchorだ。** Producerの仕事はAIが担う。

そしてArchitect/Builder——自ら手を動かしながら構造を作るエンジニア——は、AIの最高の使い手にもなる。なぜなら、構造を理解した上でAIに指示を出せるからだ。AIというProducerに、正しい方向を示せる。

## AGI——この前提すら覆るとき

ここまで、「AIはArchitectになれない」という前提で書いてきた。

しかし正直に向き合わなければならない問題がある。

**汎用人工知能（AGI）がArchitectの能力を獲得したとき、何が起きるか。**

### AGIがArchitectになれる条件

現在のAI（LLM）がArchitectになれない理由を、改めて整理する：

1. **文脈の欠如** — ドメイン知識、チームの歴史、ビジネスの方向性を統合的に理解できない
2. **長期的一貫性の欠如** — セッションを超えた設計の一貫性を保てない
3. **トレードオフ判断の欠如** — 正解のない判断（パフォーマンス vs 保守性、柔軟性 vs 単純性）にドメイン固有の重み付けができない

逆に言えば、**これらすべてを獲得したAIは、Architectになり得る。**

AGIが企業のドメイン知識を完全に理解し、チームの文脈を記憶し、ビジネスの方向性に基づいてトレードオフ判断ができるようになったとき——そのAGIは構造を設計できる。重力を生める。

### そのとき、エンジニアに何が残るか

これは悲観的な結論になるかもしれない。しかし逃げずに考えたい。

AGIがArchitectになれる世界では、エンジニアの「技術的な」価値はほぼゼロになる。コードを書く必要もなく、構造を設計する必要もない。

しかし——**ここが重要だ**——一つだけ、AGIにも代替できないものがある。

**「何を作るか」の決定だ。**

「なぜこのプロダクトが必要なのか」「ユーザーの本当の痛みはどこにあるのか」「このビジネスはどこに向かうべきか」——これは技術の問題ではない。人間の意志の問題だ。

宇宙の比喩を続けるなら：

```
星を作れる         ← AI (Production)
重力を作れる       ← AGI (Design)
宇宙を作る理由を持てる ← 人間 (Purpose)
```

AGIは宇宙を構築できるかもしれない。しかし「なぜその宇宙を作るのか」を決めるのは、人間だ。

### どこまで行っても「楽しい」を決めるのは人間

私は今、エンターテインメントの領域にいる。

なぜこの領域を選んだか。それは**「楽しい」を定義する行為は、人間にしかできない**と考えたからだ。

エンタメとは、人間が「これが楽しい」と決める行為だ。AIがどれだけ進化しても、「楽しい」の基準を決めるのは人間だ。ゲームのルールを作ること。体験を設計すること。「この瞬間が面白い」と感じる感覚を理解し、それをプロダクトに落とし込むこと。

これはPurposeの最も具体的な形だと思っている。AGIがコードを書き、構造を設計できる時代が来ても、**「どこまで行っても楽しいを決めるのは人間」**——この事実は変わらない。

だから私はこの領域に飛び込んだ。そしてこの確信は、AI時代を生きるエンジニアの一つの答えでもある。技術の上に、人間の意志を乗せる。それが、どの時代にも廃れない仕事だ。

### 楽観でも悲観でもなく

私は技術的特異点（シンギュラリティ）を、楽観でも悲観でもなく、**事実として**見ている。

AGIが来るかどうかは分からない。来るとしてもいつかは分からない。しかし、その可能性を無視して「エンジニアの価値」を語るのは不誠実だ。

今この瞬間、私にできることは明確だ。**今のAIの能力を正確に理解し、人間にしかできない仕事に集中すること。** そしてAIの能力が拡張されるたびに、「人間にしかできないこと」の定義を更新し続けること。

これは退却戦ではない。**最前線の再定義だ。**

## AIコードの考古学

git考古学のもう一つの使い方が見えてくる。

**AIが書いたコードと人間が書いたコードのSurvival差を追跡する。**

もしAIが書いたコードのSurvivalが系統的に低いなら、それはAIが構造に合わないコードを生成している証拠だ。逆に、AI生成コードのSurvivalが人間と同等なら、Architectが設計した構造がAIにとっても十分に明確——つまり、構造が「文明の法典」として機能していることの証拠になる。

これは第14章のCivilization Testの変形だ。

> **Architectが去った後、文明は続くのか？**

これがCivilization Testだった。AI時代のバリエーションはこうだ：

> **AIがコードを書いても、構造は維持されるのか？**

AIは、Architectが去るのとは違う形で、構造にストレスをかける。Architectの不在は構造の放棄だが、AIの介入は構造の**希釈**だ。大量の星が生まれることで、重力場が薄まっていく。

このストレスに耐えられる構造だけが、AI時代の文明になれる。

## AI時代のArchitect/Builder

第13章で、私はArchitect/Builderでありたいと書いた。

AI時代において、この志向はさらに重要になる。

Architect/Builderは**構造をコードで示す**。設計のドキュメントではなく、実装そのもので「こう書くべきだ」を示す。

これはAIにとっても同じだ。AIがコードを生成するとき、既存のコードベースを参照する。Architect/Builderが残した実装パターンは、AIにとっての**構造的テンプレート**になる。

つまりArchitect/Builderの仕事は、人間の後続エンジニアだけでなく、**AIに対しても設計の意図を伝える**ことに拡張される。

良い設計とは、人間が読んでもAIが読んでも、同じ方向にコードが書かれる構造だ。

### AIへの魂の継承

ここで、前章の文明の方程式がAI時代にどう機能するかが見えてくる。

**Architectが基盤を作る。** モジュール境界、依存の方向、命名規則——コード宇宙の重力場を定義する。

**Anchorがその基盤の上で、AIに構造を理解させる。** CLAUDE.mdやアーキテクチャドキュメントを整備し、AIが構造に沿ったコードを生成できるよう、適切な知識を与える。エントロピーが増大しないよう、AIの出力を構造の文脈で検証し、軌道修正する。

そしてArchitectとAnchorが互いを補完しながら、**AIという最強のProducerとともに、秩序を持って大量の機能をプロダクトに紡いでいく。**

これは第14章で書いたSuccession——構造的知識の継承——の新しい形だ。人間から人間への継承ではない。**人間からAIへの継承。** Architectが設計した構造の魂を、AnchorがAIに伝える。AIはその魂を受け取り、構造に沿ったコードを生む。

いわば**AIへの魂の継承**だ。

文明が世代を超えて続くように、コードベースの構造もAIという新しい「世代」に継承される。CLAUDE.mdは、その文明の法典になる。AIが読み、理解し、従うべきルール。人間のArchitectが込めた設計の意図が、AIの手を通じて実装され続ける。

**文明は人間だけで紡ぐものではなくなった。** 人間とAIが共に紡ぐ時代が、もう始まっている。

## 私はAIをどう使っているか

正直に書く。

この14章分のブログ記事は、AIと一緒に書いた。EIS自体も、Claude Codeとの壁打ちから生まれた。私は日常的にAIをPair Programmingの相手として使っている。

しかし、**構造の判断は人間がしている。**

AIに「この機能を実装して」と頼むことはある。しかし「このシステムをどう構造化するか」は自分で決める。モジュール境界、依存の方向、命名規則、アーキテクチャパターン——これらの設計判断はAIに委ねない。

AIは最高のProducerだ。しかしArchitectは人間がやる。少なくとも今は。

そしてそれこそが、AI時代のエンジニアとして最も鍛えるべき筋肉だと思っている。

**コードを書く筋肉ではない。重力を生む筋肉だ。**

## おわりに——星の爆発の時代に

宇宙の歴史には、星の形成が爆発的に増えた時代がある。**スターバースト**と呼ばれる現象だ。銀河が衝突したとき、ガスが圧縮され、大量の星が一気に生まれる。

AI時代は、コード宇宙のスターバーストだ。

星は爆発的に増える。しかし、重力がなければ銀河は形を保てない。星がバラバラに飛散して終わる。

だから今、重力を生む人間が必要だ。

構造を作れるエンジニア。設計を実装で示せるエンジニア。AIが書いたコードに秩序を与えられるエンジニア。

git考古学の望遠鏡は、その重力を可視化するために作った。

AI時代にこそ、望遠鏡を覗いてほしい。あなたのコード宇宙に、重力はあるか。
