---
title: "第5章 —— プロダクトと組織の同型性"
---

*プロダクトの UX が崩れている場所には、組織の同じ場所に空洞がある。プロダクトと組織は対称に壊れ、対称に直る。*

---

> **本章の射程**：思想レイヤー（プロダクトと組織が同型構造であることを論じる）+ 設計レイヤー（同じ語彙で両者を扱える設計原理を示す）

### こんな場面から始まる

長く運用されてきた SaaS の改善プロジェクト。ユーザーから「画面遷移が多すぎる」「同じ概念が場所によって違う名前で呼ばれている」「機能が増えすぎて自分が今どこにいるか分からない」というフィードバックが続く。UX チームが入って、再設計の会議が始まる。

数週間後、UX チームは諦め顔で言う。「画面の問題ではない。**機能の整理が必要だが、誰がそれを決められるのかが分からない**」。エンジニア組織を見渡してみると、プロダクトオーナーは複数存在し、それぞれが自分の機能群を守っている。横断する権限を持つ人がいない。

UX が崩れている場所に、組織の同じ場所が空洞になっている。**プロダクトを直そうとして、最初に直さなければならないのは組織**だった——これが本章の出発点だ。

---

## 同型性の主張

序論で「本書で『構造』が指す対象」として、コードの構造・人材の構造モデル・組織のレイヤー構造の 3 つを挙げた。本章は、ここに **4 番目の対象——プロダクトの構造**を追加する。

第 4 章で導入した **3 層 + 2 変換** モデル（原理層 ── 変換 ── 構造層 ── 変換 ── 実装層）は、そのままプロダクト側にも当てはまる：

| | 組織 | プロダクト |
|---|---|---|
| **原理層** | 戦略、世界観、思想 | テーマ、デザイン原則、UX プリンシパル、何を作らないか |
| ↕ **変換（原理 ↔ 構造）** | 思想を組織設計・アーキテクチャに翻訳／組織の現実を思想の再評価に戻す | 思想を情報設計・IA に翻訳／画面の実挙動を原則の再評価に戻す |
| **構造層** | アーキテクチャ、組織設計、運用設計 | 情報設計、画面遷移、状態モデル、機能群の組織化 |
| ↕ **変換（構造 ↔ 実装）** | 組織設計を日々の運用・コードの規約に翻訳／実装の手触りを構造の修正に戻す | 情報設計を画面・API の具体に翻訳／画面の挙動を情報設計の修正に戻す |
| **実装層** | コード、機能、運用 | 画面、操作、マイクロインタラクション、API |

そして本章の中心命題はこうだ：**この 4 番目（プロダクトの構造）は、3 番目（組織のレイヤー構造）と、3 層 + 2 変換のレベルで同型である**。

これは比喩ではない。**設計原理**だ。

- 組織の構造層が薄く変換が止まっている組織は、プロダクトの情報設計も崩れている
- プロダクトの原理層（テーマ・原則）が定まっていない組織は、組織の原理層も定まっていない
- プロダクトの実装層がバラついている組織は、エンジニアの実装層も Style の混在で動いている
- 組織で原理 ↔ 構造の変換が滞っている組織は、プロダクトでもテーマが画面設計に降りてこない

**両者は対称に壊れ、対称に直る**。

### なぜ同型なのか

理由はシンプルだ。**プロダクトは組織が作るもの**だから。

Conway's Law は「組織構造はアーキテクチャに映る」と言ったが、本書のレイヤー軸では一段強く言える：**組織の 3 層 + 2 変換は、プロダクトの 3 層 + 2 変換にそのまま写像される**。組織が原理層を語る言語を持たなければ、プロダクトの原理層は記述されないまま放置される。組織に構造 ↔ 実装の変換者がいなければ、プロダクトの情報設計と画面の整合は誰にも保たれない。

逆方向も成り立つ。プロダクトの原理層を強く設計し直すと、組織は否応なく原理層の議論を始める。**プロダクトを変えると組織が動き、組織を変えるとプロダクトが動く**。

## プロダクトの三層を観測する

組織側は EIS という観測装置を持っている。プロダクト側は何で観測するか。本章では、各レイヤーごとに観測対象を提示する。

### プロダクトの実装層

**ユーザーが直接触る画面・操作・マイクロインタラクション、API**。プロダクトが developer 向けの API ならば、API そのものが実装層の主役になる。

観測対象：
- 個別画面 / API エンドポイントの動作・レスポンス時間・挙動
- バグ報告、ユーザーサポートチケット、SDK / API インテグレーションのフィードバック
- ヒートマップ、クリック率、エラー率（UI 側）／ 5xx 率、後方互換性破壊、deprecation 通知（API 側）
- **細部の一貫性**：画面間 / API 間で**同じ操作が同じパターンで動くか**（同じアクションのアイコンが画面ごとに違わないか、API のエラーレスポンスが統一されているか）
- 画面・API ごとの「使えるか / 使えないか」のシグナル

評価軸：**動くか、使えるか、細部が一貫しているか**。組織の実装層と同じ問いだ。

### プロダクトの構造層

**情報設計、画面遷移、状態モデル、機能群の組織化**。プロダクト全体としての一貫性が問われる、思想が形を持った状態のレイヤー。

観測対象：
- 画面間の遷移グラフ（不必要な迂回がないか）
- 同じ概念が複数箇所で別の名前で呼ばれていないか
- 「ある機能が別の機能を妨げる」関係がないか
- ユーザーが現在地を把握できるか（パンくず、ナビゲーションの一貫性）
- 機能を追加するとき、**既存の情報構造のどこに置くか**で議論が起きるか（議論なく追加されるなら、原理 ↔ 構造 の変換が動いていない）

評価軸：**構造として筋が通っているか**。組織の構造層と同じく、ここは**変換の着地点**だ——原理層の思想がここで形になり、構造 ↔ 実装の変換を経て画面になっていく。構造層が薄いと、どの変換も成立しない。

### プロダクトの原理層

**世界観、テーマ、デザイン原則、UX プリンシパル、何を作らないか**。第 4 章「原理層を明文化する」で挙げた成果物群が、ここに住む。

観測対象：
- テーマ・世界観のドキュメント（あるか、最新か）
- デザインプリンシパル（明文化されているか、議論されているか）
- UX プリンシパル（優先順位の禁則、誰のために最適化するかの言語化）
- 「何を作らないか」リスト（実在するか、機能依頼に対して引用されるか）
- **思想の内部一貫性**：複数の原則が互いに矛盾していないか、機能群が同じ原則の下で設計されているか

評価軸：**この判断は何のために行われているのか説明可能か、思想として一貫しているか**。AI に委ねられない判断だからこそ、人間の判断履歴を可視化する必要がある——これは組織の原理層と同じだ。

> **テーマ・UI プリンシパル・UX プリンシパルが一つも書かれていない——そんな状態なら、明日の機能会議より先に書く必要がある**。これは「観測対象が未整備」ではなく、**判断の拠り所そのものが不在**の状態だ。原理層なしに積み上がるプロダクトは、方向を持たない運動になる。**迷走している可能性が高い**——と言い切っていい状況だ。
>
> そして、原理層は書いて終わりではない。**明文化し、全員で掲げ、議論のなかで矛盾が見つかったらその都度置き換え・整理する**——この循環こそが原理層の本体だ。書き残されたまま誰も引用しない「化石ドキュメント」は、不在と同じ症状を生む。生きた原則だけが、原理層として機能する。
>
> ただし、**重厚長大になっても機能しない**。全員で掲げ続けられる量・粒度でなければ、どれほど精緻に書かれていても現場の判断には降りてこない。**短く、引用可能で、書き換えやすい**——これが原理層の実用仕様だ。立派な思想体系よりも、**今月の意思決定に実際に効いている 1 枚の原則**のほうが、原理層として強い。
>
> 実務的な目安として、**1 つのプリンシパル集は多くて 10 個以内**に収める。これを超えるなら、**分割を検討する合図**だ——多くの場合、粒度や対象領域が混ざっている。たとえば toC と toB では、ユーザーの前提も意思決定の速度も違うため、**別のプリンシパル集として分けたほうが機能することが多い**。一つに押し込むと、どちらにも効かない抽象論になりがちだ。

### 一貫性は二軸で見る

プロダクトの一貫性は、**水平**と**垂直**の二軸で見る必要がある。

**水平の一貫性（各層の内側）**

- **実装層** → 細部のパターン一貫性（同じ操作、同じアイコン、同じ API パターン）
- **構造層** → 構造の一貫性（同じ概念、同じ名前、IA が筋を通す）
- **原理層** → 思想の一貫性（テーマ・原則が内部矛盾していないか）

**垂直の一貫性（層を貫通する変換）**

決定的に重要なのは、原理層の思想が、**構造層と実装層でそれぞれの言語に適切に変換され、なおかつ一貫しているか**だ。

たとえば原理層で「シンプルさ」を掲げているなら：

- **構造層への変換**：「探さなくて済む情報設計」「画面遷移を最小化する IA」として翻訳される
- **実装層への変換**：「ボタンが少ない」「最小操作で完了する」「API 引数を増やさない」として翻訳される

各層には固有の語彙がある。原理層の言葉をそのまま下ろしても意味は通らない——「シンプルさ」と書かれた付箋だけが下りてきても、実装者は何をどうすればいいか分からない。**各変換が起きていないと、思想は層をまたいだ瞬間に消える**。

逆方向も重要だ。構造や実装での決定が、原理層に**戻れる**こと。「この画面遷移は、原理層の何に対応する変換か？」が説明できないなら、その決定は原理層から切り離されている。

第 4 章で扱った**変換者**は、まさにこの垂直の一貫性を担保する人だ。原理を構造と実装の語彙に翻訳し、実装で起きた現実を構造を経由して原理に戻して再評価する——これが変換者の核心的な仕事になる。

垂直の一貫性が崩れているプロダクトは、**原理層が形骸化している**。書かれていても、各層の語彙に変換されないまま放置されている。これは組織側の症状でもある——戦略は掲げられているが、現場の決定に降りてきていない状態だ。**垂直の一貫性（= 2 つの変換がちゃんと動いているか）は、プロダクトと組織の両方で観測すべき軸**になる。

## 同型性の症状

組織のレイヤー不一致が「シニアを増やしたのに前に進まない」「戦略がコロコロ変わる」として現れたように、プロダクト側にも対称の症状がある。

### 「機能は増えているのに、ユーザーが迷う」

**プロダクト側の症状**：機能 1 つずつは正しく動く。しかし全体としては、ユーザーが「何ができるか」「自分はどこにいるか」を把握できない。

**組織側の対応症状**：プロダクトオーナーが分散していて、横断する変換者がいない。**構造層が薄く、原理 ↔ 構造の変換が止まっている**。

**介入**：プロダクト側で情報設計を作り直すだけでは解けない。**組織側に原理 ↔ 構造の変換者を配置する**——思想をプロダクトの情報設計に落とし、構造の歪みを原則に戻せる人だ。これによりプロダクトの構造層を継続的に整理できる体制ができる。

### 「画面ごとにデザインが違う」

**プロダクト側の症状**：機能ごとに別のデザイナーが担当し、原則のすり合わせがされない。同じプロダクトの中で言語が違う。

**組織側の対応症状**：原理層に住む人がいない、または**原理 ↔ 構造の変換が止まっている**ため、思想が構造層に下りてこない。デザインプリンシパルが書き留められていない。

**介入**：**コンポーネントライブラリやデザイントークン（実装層の具体物）**を整えるだけでは、原理層の空洞は埋まらない——部品が揃っても、それを貫く思想が書かれていなければ、各デザイナーは各自の原則で動き続ける。**組織側で原理層の言語化を継続的に回す**——テーマやプリンシパルを書く人、それを使い続ける人、書き換える人を配置する。デザインシステムは、**原理層の言葉（テーマ・原則）と実装層の具体物（コンポーネント）が揃って初めて機能する**——片方だけでは足りない。

### 「リファクタしたいのに、何が要るのか合意できない」

**プロダクト側の症状**：技術的負債は明らかにある。直したい意思もある。しかし「何を、どの順序で、どう直すか」の合意ができない。

**組織側の対応症状**：プロダクトの原理層が未定義のため、「**この負債は何のために返すのか**」が言語化できない。判断軸がない。

**介入**：技術的議論で詰めるよりも先に、**プロダクトの原理層を整理する**——テーマと優先順位を明文化することで、技術的負債の優先順位が自動的に決まる。

### 「同じ操作なのに、画面や API によって挙動が違う」

**プロダクト側の症状**：同じ「削除」でも画面によって確認ダイアログが出たり出なかったり、同じ概念なのに API によってキー名やエラー形式が違う。コンポーネントは重複し、エッジケースの扱いもばらつく。**細部のパターンが揃っていない**。

**組織側の対応症状**：実装層のレビュー文化が薄く、Style を揃える習慣がない。共通コンポーネントや規約の責任者が不在のまま、各自が自分の領域で最適に書いている。**実装層の散乱**。

**介入**：コンポーネントライブラリやデザイントークンを整備することが出発点にはなる——ただし、**「揃える習慣」を組織に根づかせる**までが介入の本体だ。レビュー基準、共通規約、ペアプロの機会など、実装層の Style を日々揃える文化がないと、道具だけ置いても使われない。

## 介入設計の対称性

レイヤーの空洞を埋める介入も、両側で同型に設計できる。

| 症状 | プロダクト側介入 | 組織側介入 |
|---|---|---|
| 原理 ↔ 構造の変換が止まっている | 情報設計の責任者を 1 人立てる | 原理 ↔ 構造の変換者を 1 人立てる |
| 構造 ↔ 実装の変換が止まっている | デザインシステムの運用者を置く | 構造 ↔ 実装の変換者（Tech Lead 等）を置く |
| 原理層の不在 | テーマ・原則・禁則を文書化 | 戦略・優先順位・「やらないこと」を文書化 |
| 実装層の散乱 | コンポーネントライブラリ・デザイントークン | コードベース規約・テスト基盤・ドキュメント標準 |

注意：**両側に同時に介入する**のが原則だ。プロダクト側だけ直しても、組織側で再生産される構造的問題が残っていれば、半年後には同じ崩れが戻ってくる。組織側だけ直してもプロダクトに反映する経路がなければ、ユーザーには変化が見えない。

**UX 改善と組織改善は、別プロジェクトではなく、同じ設計タスクの両面**。両者を同じ会議で、同じ語彙で進められる組織だけが、本当の意味で構造を直せる。

## 現場で何が変わるか

プロダクトと組織を同型のレイヤー構造として捉え直すと、以下が変わる：

1. **プロダクト改善の議論に、組織側の責任者が参加する**。「ユーザーが迷う」を直すために、UX デザイナーだけでなく組織側のマネジメントも会議に入る——両側を一緒に直すため
2. **組織変更の前に、プロダクトのレイヤー診断をする**。「組織を再編する」前に、「プロダクトのどこが歪んでいるか」を観測する。多くの組織変更は、プロダクト側の症状を読まずに行われ、結果としてプロダクトに対する効果がない
3. **デザイン原則と戦略原則が同じドキュメントで管理される**。「プロダクトの原理層」と「組織の原理層」は、本質的には同じ意思決定記録。別々のページで管理する理由はない
4. **UX デザイナーが組織設計に発言権を持つ**。プロダクトの構造層を整理する仕事は、組織の構造層の整理と同型。UX 設計者は、本質的に**組織設計者**でもある
5. **「機能を追加する」議論が、自動的に「組織のどこが受け持つか」議論になる**。プロダクトの構造層に新しい機能を加えるとき、組織側でその責任を持つ単位が決まっていなければ、追加してはいけない

## 次章への接続

ここまで、組織を 3 軸トポロジー × 3 層 + 2 変換で記述し、さらに**プロダクトもまったく同じ 3 層 + 2 変換で動いている**ことを示した。観測の対象（コード、人、組織、プロダクト）が揃った。

しかし、観測した上で**何をするか**——介入の設計は別の問題だ。1on1、ペアプログラミング、コードレビュー、組織変更——これらの介入を、本書の三層フレームの上にどう乗せるか。

次章では、**介入設計**を扱う。行動 / 出力 / 構造の 3 層で介入を分解し、感情ではなく構造で意思決定する語彙を組む。
