---

> **この章のスコープ（骨子）**
>
> 介入は「行動 / 出力 / 構造」の 3 層で設計する。感情ではなく構造で刺す。1on1・ペアプロ・レビューをこの3層に分解して再構成する。

---

## 1. 介入の3層

- **行動層** — 何をしているか。会議、コミット、レビュー、コード時間
- **出力層** — 何が生まれたか。機能、ドキュメント、決定事項
- **構造層** — 何が残ったか / どう繋がっているか。EIS 指標、レイヤー配置

介入は必ず**どの層に対して行うか**を明示する。混ぜない。

## 2. 1on1 を再設計する

従来：「最近どう？」「困ってることある？」から始まる情緒 first の構造
再設計：最初の15分を**構造層の共有**に使う

```
冒頭：構造の共有
  ・直近3ヶ月のEISシグナル
  ・配置されているレイヤー（実装層 / 構造層 / 原理層）と担っている変換
  ・チームにおける役割（Anchor / Producer / Cleaner …）

中盤：出力層
  ・その構造に至ったプロセスの振り返り

終盤：行動層 + 情緒
  ・次の行動、困っていること、疲労、モチベーション
```

- 情緒を後ろに置く理由：**情緒を構造と切り分けた方が、情緒そのものを丁寧に扱える**
- 情緒を軽視しているのではなく、混ぜないことで**両方が機能する**

## 3. ペアプロを再設計する

- 目的の明示：「Anchor 候補の育成」 / 「Cleaner の継承」 / 「Producer の速度維持」
- ペアの組み合わせは、**個人の能力**ではなく**レイヤー移動**を意図して組む
- 例：実装層の強い Producer × 構造 ↔ 実装 の変換を担う Anchor で、Producer に構造 ↔ 実装 の変換能力を継承する

## 4. コードレビューを再設計する

- レビューは行動層ではなく**構造層**に属する介入
- 「LGTM」 ではなく「この変更は Robust か / Design 層に寄与しているか / Cleanup か」で答える語彙
- 「Nits」と「構造的懸念」を視覚的に分離

## 5. ひまのあケース（予定）

- 「能力はあるのに空転している」状態を、3層で分解する具体例
- 行動：活発。出力：薄い。構造：Breadth↑ Survival↓ Design↓
- 介入はレイヤー配置から変える。感情的な「頑張って」の声かけは、構造問題を悪化させる

## 6. 現場で何が変わるか（予定）

- 1on1 後の疲労が減る（情緒と構造が混ざらないため）
- 介入の効果が測定可能になる（次の観測サイクルで検証できる）
