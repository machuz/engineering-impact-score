title: "第6章 —— 介入設計（1on1 / ペアプロ）"
---

*その 1on1 を半年続けても、組織は 1 ミリも動かない——介入が「行動」に閉じているからだ。[行動層での慰めは、内圧の熱を点け直せない](https://library.orbitlens.io/psychological-os/#ch2)。「最近どう？」から始める時点で、その会話は蓄積を 1 グラムも動かさない。*

*情ではなく構造で刺す——それだけが、組織の蓄積に届く介入になる。*

---

> **本章の射程**：設計レイヤー（介入を「行動 / 出力 / 構造」の 3 層で分解する）+ 現場レイヤー（1on1・ペアプロ・コードレビュー・組織変更を再構成する）

### こんな場面から始まる

毎週水曜の 1on1。EM が訊く。「最近どう？」。エンジニアは少し考えて「まあ、いろいろあるんですけど……頑張ってます」と返す。EM が「何か困ってることある？」と続ける。「ええ、まあ、前の PR のレビューがなかなか通らなくて、モチベーション的にはちょっと……」「それは大変だね。気にしないでいいから、今週はゆっくり進めよう」。

会話は柔らかい。心配もしている。次の 1on1 もちゃんと入れる。それでも、**半年後、同じエンジニアが同じ詰まりを報告し続けている**。

このやり取りに何が欠けているのか。**介入が「行動層」に閉じている**からだ。気遣いも声かけも次の予定も、すべて「今週の行動」に働きかけている。その人が出しているアウトプットの質や、何が残り続けているか——**出力層**や**蓄積層**には一度も触れていない。

---

## 介入の 3 層

![介入の 3 層 — 蓄積が行動と出力の条件を決める](https://raw.githubusercontent.com/machuz/eis/main/docs/images/blog/sdo-ch6-three-intervention-layers.svg)

組織に対する介入は、働きかける対象によって 3 層に分かれる——**行動**（何をしているか）、**出力**（何が生まれたか）、**蓄積**（何が残り、後の判断と実装に効いてくるか）。

ここが構造駆動の核心だ——**この 3 層は対等ではない**。**蓄積が、行動と出力の条件を決めている**。何が残っているか／どう繋がっているかが、現場の人間の行動を規定し、そこから生まれる出力の質を決める。だから行動層だけに介入しても、蓄積が変わらなければ、行動は放っておけば元に戻る。

逆から読めば、こうなる：**蓄積を変えれば、行動と出力は勝手に変わる**。これが本書が「構造駆動」と呼んでいる因果の向きだ。従来の介入が失敗するのは、因果を逆向きに走らせているからだ——行動を変えて、出力が出るのを期待して、そのうち構造も変わるだろう、と。順序が逆だ。

### 行動層

**その人が日々何をしているか**。

- 会議、コミット、レビュー時間、Slack での発言、デプロイ頻度、オンコール時間
- 観測しやすく、変えやすく、すぐ効果が測れる
- しかし——**行動だけを変えても、出力や構造には残らない**

「ミーティングを減らそう」「Slack で気軽に話そう」「1on1 を増やそう」——これらはすべて行動層の介入だ。有効な場面はあるが、これだけで組織が動いた試しは少ない。

### 出力層

**その人から何が生まれたか**。

- 機能、ドキュメント、意思決定、RFC、設計提案、メンタリングの形跡
- 行動より一段重い
- しかし——**出力も、使われなければ構造にならない**

書かれた RFC が誰にも引用されない、出した機能がリリース後に使われない、議事録が参照されない。**出力があっても、それが後に続く意思決定や実装に繋がらなければ、構造には積み上がらない**。

### 蓄積層

**何が残り、どう繋がっているか**。

- EIS の 7 軸指標、レイヤー配置、変換経路、共通規約、チーム構成、採用基準
- 最も重い。変えにくい
- しかし——**ここが変わると、行動と出力は自動的に変わる**

蓄積層は、**個別の行動や出力が積み上がった結果として現れる状態**であると同時に、**次の世代の行動と出力の条件を決める土台**でもある。直接は触れにくいが、蓄積層が変わった瞬間、そこに繋がる全ての行動と出力の意味が変わる。

介入を構造駆動で設計するとは、**「行動をどう変えるか」ではなく「何を蓄積させるか」から考える**ことだ。

### 介入の原則：層を混ぜない

**介入は、どの層に対して行うのかを明示する**。混ぜると効かない。

- 「コミュニケーションを良くしよう」 → 行動・出力・蓄積が混じっている。どこに効くのか誰にも分からない
- 「**1on1 の頻度を増やす**」 → 行動層
- 「**ドキュメントを残すルールを作る**」 → 出力層
- 「**レイヤー配置を組み直す**」 → 蓄積層

曖昧な介入は「なんとなく効いた気がする」で終わる。層が指定された介入は、**次の観測サイクルで効いたかどうか検証できる**。介入を構造にするには、まず介入を明示するところから始まる。

## 1on1 を再設計する

従来の 1on1 は情緒 first で始まる——「最近どう？」「困ってることある？」。この入口の時点で、**介入が行動層に固定される**。情緒の共有は行動だ。構造の話には自然には降りていかない。

再設計：**構造 → 出力 → 行動 + 情緒** の順で進める。

```
冒頭（15 分）：蓄積層の共有
  ・直近 3 ヶ月の EIS シグナル
    - Production / Survival / Design / Cleanup / Breadth / Quality / Indispensability
  ・配置されているレイヤー（実装層 / 構造層 / 原理層）と、担っている変換
  ・チームにおける役割（Anchor / Producer / Cleaner / Specialist …）

中盤（15 分）：出力層
  ・直近の主な出力と、その構造に至ったプロセスの振り返り
  ・書いた RFC、設計判断、メンタリング、レビュー内容

終盤（15 分）：行動層 + 情緒
  ・次の行動、困っていること、疲労、モチベーション
  ・前半 30 分で見えた構造を踏まえた上で話す
```

### なぜこの順序なのか

情緒を**後ろ**に置く理由は、情緒を**軽視しているから**ではない。むしろ逆だ——**情緒を構造と切り分けたほうが、情緒そのものを丁寧に扱える**。

情緒 first で始まると、その場の感情が 1on1 全体の色を決める。「今日は元気がないから、構造の話は後日にしよう」が起きる。結果、**構造の話は永遠に後日になる**。

構造 first で始まれば、情緒が構造の上に乗る。「コードは残っている。Role も Anchor に近づいている。ただ、最近疲れていて、眠れていない」。このフレーミングは、同じ疲労の訴えでも**意味がまったく違う**。情緒は「別の問題」ではなく「構造のなかの現象」として扱える。

> **観測の現場から**
>
> これは何度も見てきたパターンだ——毎週 1on1 を 30 分やっている。1 年続けた。それなのに、そのエンジニアのキャリア観についても、組織内での立ち位置についても、EM と本人で**ほぼ何も合意が進んでいない**。
>
> 理由はシンプルだ。1on1 のたびに、話題が「今週の課題」「今の気持ち」に戻る。**構造の話が 1 回も出ないまま、1 年分の時間が消費された**。
>
> 情緒は貴重だ。しかし情緒だけを 1 年扱っても、キャリアは動かない。構造を扱う時間を先に確保することでしか、情緒の扱いも結果的に深まらない。

### 観測は評価ではなく、理解のため

構造を 1on1 の冒頭に置く、と書くと、**監視の装置にしたいわけではない**、とどうしても言いたくなる。逆だ。構造を先に開く意味は、「**相手がどの方向に時間とエネルギーを投じてきたか**」をまず理解することにある。

- Production が低い月がある——**怠けた**のではなく、Design や Survival の試行錯誤に時間を使っていたかもしれない
- Survival が低い期があった——**質を落とした**のではなく、速さを求められた状況の必然だったかもしれない
- Breadth が広く Design が薄い——**散らばっている**のではなく、複数ドメインの探索段階で意図的にそうしているかもしれない

**シグナルだけで判断してはいけない**。シグナルは、その人が**頑張ってきた方向**を読み取るための地図だ。方向が理解できて初めて、介入はその人の努力と噛み合う形で設計できる。方向を理解せずに入る介入は、上から抑え込む評価の延長にしかならない——**本書が最も避けたい姿だ**。

1on1 で構造を先に置くのは、「あなたは何をしているのか」を確認するためではなく、「あなたが何に向き合ってきたかを、私はちゃんと見ている」と伝える行為だ。

### 自分ごと化の問い

> 先週、あなたが入った 1on1 のうち——
>
> - **15 分以上、蓄積層の話をした 1on1** は何件あっただろうか？
> - 相手の直近 3 ヶ月の**出力**について、具体的に言及できただろうか？
> - 「最近どう？」で始まった 1on1 が、**最後まで行動層に閉じて終わった**ことはなかっただろうか？

### 明日、これだけ試してほしい

> 次の 1on1 で、**最初の 15 分を「蓄積の話」だけ**に使う。
>
> - EIS があれば、その人の直近 3 ヶ月のシグナル（Production / Survival / Design / Cleanup / Breadth / Quality / Indispensability）をそのまま共有する
> - EIS がなければ、「この人は 3 ヶ月で**何を残したか**」「その出力は**どこで参照されているか**」だけを問う
> - 情緒・困りごと・次の行動は、30 分後に話す
>
> それだけで 1on1 の意味は変わる。1 回の試行で、構造が会話に乗るかどうかの手応えが取れる。

## ペアプロを再設計する

従来のペアプロは「能力差でペアを組む」——先輩が後輩に教える、熟練者が初心者に伴走する。これは**行動層の介入**だ。熟練者の手さばきを見せる、質問に答える、コード時間を共有する。

再設計：**ペアの目的を、レイヤー移動として明示する**。

- 「**Anchor 候補の育成**」 → Anchor 型の先輩 × 素養のある若手。構造 ↔ 実装 の変換能力を継承する
- 「**Cleaner の継承**」 → 既存の Cleaner × 負債に立ち向かえる若手。清掃の型を伝える
- 「**Producer の速度維持**」 → 2 人の Producer をペアに。互いのコード速度に引き摺られて速度が保たれる
- 「**変換能力のスケール**」 → 構造 ↔ 実装 の変換を担う Anchor × 実装層の強い Producer。Producer に変換能力を install する

ペアの組み合わせは、**個人の能力**ではなく**レイヤー移動の意図**で組む。意図が明示されたペアプロは、**蓄積層の介入**になる——その人の型が変わる、役割が変わる、チームの配置が変わる。

## コードレビューを再設計する

多くの組織で、コードレビューは行動層の介入になっている——「LGTM」「Nits: typo」「ここは好みですが…」。反応時間、コメント数、LGTM の速さが計測され、それで「レビューが活発なチーム」と評価される。

しかし本来、**コードレビューは蓄積層の介入**であるべきだ。

### レビューの語彙を再設計する

「LGTM」の代わりに、**構造軸での評価**を明示する：

- **Design**：この変更は、コードベースの Design 層（アーキテクチャの中心）に寄与しているか、表層の utility に留まるか
- **Survival**：この変更は Robust に残りそうか、それとも 3 ヶ月以内に書き換わる shortcut か
- **Cleanup**：これは他者の負債を清掃しているか、自分の意図だけの書き換えか
- **Quality**：この変更が fix を生む可能性はどの程度か

この語彙でレビューすると、**レビュー自体が蓄積層の観測記録**になる。EIS のシグナルと照らし合わせたとき、「あの Anchor のレビューは、実際に Design 層への寄与を高めている」という構造的な評価ができる。

### 「Nits」と「構造的懸念」を視覚的に分離

全ての指摘を並列に扱うと、**nit が構造的懸念を埋める**。レビューの疲労の大半はこれだ。

- 🟢 **Nits**：スタイル、命名、軽微な改善提案。ブロッキングしない
- 🟡 **Non-blocking suggestion**：議論する価値はあるが、この PR では見送れる
- 🔴 **Blocking**：構造的懸念。Design / Survival / Quality に影響する

視覚的に分離するだけで、レビューは**行動層から蓄積層へ**昇る。

## 組織変更を再設計する

「組織を再編する」という意思決定は、多くの場合ツリー図のリファクタリングで終わる——箱の位置を動かし、レポートラインを引き直し、新しい肩書を配る。これは**出力層の介入**だ。新しい組織図は出力、その時点で評価できる。しかし**構造は何も変わっていない**。

再設計：**組織変更の目的を、蓄積層の変化として定義する**。

- 組織図を書き直す前に、**レイヤー図**を書く——誰がどの層に住み、どの変換を担っているか
- 組織変更の目的を**層の薄さ・変換の不在**として定義する。「構造層が薄い」「原理 ↔ 構造 の変換が止まっている」
- 箱を動かすのではなく、**変換者を置く／コード接続を残す／役割を分離する**——何が変わることを期待しているのかを構造語彙で書く

多くの組織変更は、このマッピングをしないまま行われる。結果、箱は動いたが変換者は同じ場所で詰まったまま——**組織図だけ変わって、構造は動いていない**。これは第 4 章の「『実装は早いのに方針がブレる』」や「『シニアを増やしたのに前に進まない』」症状の親玉だ。

## 「優秀なのに空転する」を再介入する

第 3 章と第 4 章で触れた「能力は高いのに空転する人」のパターン。これを 3 層で分解すると、介入の設計が変わる。

よくある観測：**行動は活発、出力は薄い、構造は Breadth ↑ / Survival ↓ / Design ↓**。

- **行動層** — 会議参加数、発言数、レビュー反応は平均以上
- **出力層** — 書いたコード量はあるが、機能として定着していない
- **蓄積層** — EIS では Breadth だけ高く、Survival と Design が低い（広く浅く散らばっている）

この蓄積を見ずに「頑張って」「もう少しフォーカスして」と声をかけるのは、**行動層の介入**だ。本人は既に行動量では十分に頑張っている。行動を増やしても、蓄積の散らばりは縮まない。

正しい介入は**蓄積層**にある：

- **レイヤーの配置を変える**：Breadth が過剰なので、特定のレイヤー／ドメインに配置し直す
- **型を変える**：散らばっている Producer を、Specialist として一領域に座り直させる
- **ペアで変換能力を install する**：その人に、構造 ↔ 実装 の変換を扱える相手とペアを組ませる

**人を変えるのではなく、配置を変える**——第 4 章で扱った原則が、ここで具体的な介入手段になる。

## 現場で何が変わるか

介入を 3 層で分解する語彙を組織に入れると、以下が変わる：

1. **1on1 の疲労が減る**。情緒と構造が混ざらないため、両方を丁寧に扱える
2. **介入の効果が測定可能になる**。次の観測サイクル（EIS の次回実行）で、構造シグナルが変わったかを確認できる
3. **「人の問題」から「配置の問題」へ言語が切り替わる**。「A が空転している」ではなく「A は Breadth ↑ / Survival ↓ の Spread 型で、蓄積が薄く散らばっている」と言えるようになる
4. **組織変更の前にレイヤー図を書く文化が生まれる**。箱を動かす前に、誰がどの変換を担うべきかを合意できる
5. **レビューが蓄積層の観測記録になる**。LGTM の数ではなく、Design 層への寄与が観測できる

介入を感情ではなく構造で刺せるようになる——これが本章の到達点だ。

> **1on1 を何年やっても組織が変わらないのは、人が悪いからでも、才能が足りないからでもない。介入がずっと行動層に閉じていて、蓄積に手を入れなかったからだ。**

## 次章への接続

ここまで、組織を観測し、構造を記述し、介入を設計する語彙を整えた。しかし、個々の介入が**個人の解釈力に依存するかぎり**、組織は介入者の世代交代で元に戻る。

介入が**組織の言語**になったとき、それは**文化**になる。文化になれば、介入者が変わっても構造が維持される。

次章では、**文化の作り方**を扱う——本書の語彙（EIS、3 層、変換、Role × Style × State）が、組織の日常の会話に定着するまでのプロセス。文化は価値観の共有ではなく、**言語の共有**から生まれる。
