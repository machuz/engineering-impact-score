*能力は内側にあるから見えない。構造への寄与は外側に残るから見える。人を「能力」ではなく「3 軸の型」で捉え直す。*

---

> **本章の射程**：設計レイヤー（人を観測可能な 3 軸の型として記述する語彙を組む）

## 「能力」で捉えることの限界

エンジニア組織は、人を「能力」という語で論じがちだ。

- 「設計能力が高い」
- 「問題解決能力がある」
- 「コミュニケーション能力が足りない」

これらは日常語として自然で、そのぶん**何を論じているかが曖昧**になりやすい。能力はその人の内側にある属性だ。内側にあるものは、外からは観測できない。観測できないものを前提にした議論は、結局は「印象の持ち寄り」になる。

本書では、能力で人を捉えることをいったん保留する。代わりに、**構造への寄与の型（archetype）**で人を捉える。型は、過去のコードと git 履歴から読み取れる。観測可能だから、会話に乗る。

強調しておきたいのは、型は**性格分類ではない**ということだ。MBTI や 16Personalities のような「本人の本質」を当てるものではない。**ある時期、ある組織、あるレイヤーで、その人がどう構造に寄与したか**の観測結果だ。組織が変われば型も変わる。フェーズが変われば型も変わる。

型は固定ラベルではなく、**観測スナップショット**である。

## 3 軸トポロジー：Role × Style × State

人を 1 つの型に押し込めるのは、情報を潰す。EIS は、人材を**独立した 3 つの軸**で記述する：

| 軸 | 問い | 種類 |
|---|---|---|
| **Role** | 何を貢献しているか | Architect / Anchor / Cleaner / Producer / Specialist |
| **Style** | どう貢献しているか | Builder / Resilient / Rescue / Churn / Mass / Emergent / Balanced / Spread |
| **State** | いまどのライフサイクル段階か | Former / Silent / Fragile / Growing / Active |

1 人のエンジニアは、**（Role, Style, State）の 3 つ組**で表現される。

たとえば：

- **Architect / Builder / Active** —— 設計を生み出し、積極的に骨格を積み上げ、現在も活動中の人
- **Anchor / Mass / Active** —— 品質で守りたい意思はあるが、書いたコードが残っておらず、それでも今も働いている人
- **Producer / Churn / Active** —— 量は出しているが書き換え率が高く、現在も活発な人

同じ Role でも Style と State が違えば、組織への影響はまったく違う。**軸を混ぜないこと**が最初の原則だ。

以下、3 軸をそれぞれ分解する。

## Role —— 何を貢献しているか

コードベースという生態系は、**3 つの貢献種**を必要とする：

- **Architect（創造）** —— 構造を**新たに生む**。地形を形づくり、他者が伸びる環境を作る
- **Anchor（維持）** —— 構造を**守る**。土壌を安定させ、生態系の崩壊を防ぐ
- **Producer（拡張）** —— 構造の上で**機能を広げる**。ユーザー価値を生む

このいずれかが欠けると、コードベースは壊れる。Architect だけなら構造はあるが機能がない。Producer だけなら機能はあるが構造が腐る。Anchor がいなければ、構造も機能も時間とともに腐敗する。

Role にはさらに 2 つ、派生的な貢献型が加わる：

- **Cleaner（清掃）** —— 他者の負債を清掃することで構造を守る派生型。Anchor に近い
- **Specialist（深耕）** —— 一領域に深く、他に広がらない。特定領域の Anchor

以下、それぞれを短く観測像で押さえる。

### Architect

**構造を新たに生み、他者のコードにも設計方針を引き継がせる人。**

- シグナル：Design ↑、Robust Survival ↑、Breadth ○
- 現場像：
  - 自分の書いた行だけでなく、自分が書いていない行にまで設計が浸透している
  - 「この部分はどうしてこの構造なのか？」と問われたとき、組織の誰かが答えられる状態を作る
  - アーキテクチャドキュメント、RFC、設計レビューで軸を通す
  - 組織の**重力そのもの**を作る

#### Architect の 2 系譜：Inheritance と Emergent

Architect はひとつの型ではない。**進化の出自が 2 つある**。

**Inheritance Architect（継承型）** —— Anchor から進化する

- 既存構造を深く理解している
- 実世界の制約を知っている
- 破壊より洗練が得意
- システムを**壊さずに強化する**

**Emergent Architect（創発型）** —— High-Gravity Producer から進化する

- 既存構造を継承せず、**新しい構造を生む**
- 初期は摩擦が高い（他者と衝突する、書き換えられる）
- しかし最終的には**新しい重力の中心**を作る

この 2 系譜は「その人がどの道を通って Architect になったか」という**出自**を表す。Role 軸の話だ。

どちらの系譜が強く求められるかは領域による：バックエンドは**ドメイン境界が明確で正解に近い設計が存在する**ため Inheritance、フロントエンドは**唯一の正解がなく構造提案のぶつかり合いが必要**なため Emergent——がラフな近似になる。

#### 出自は固定、姿勢は混ざる

ここで明示しておきたいのは、**出自と姿勢は別のレイヤー**だということだ。

系譜（出自）は Role 軸の話で、その人がどう Architect になったかを示す。一方、**Architect が日々の仕事で取る思考の姿勢**は Style 軸の話であり、Role とは独立に観測される。本書が 3 軸トポロジー（Role × Style × State）を独立に扱う意味がここにある。

姿勢は 2 種類に分解できる：

- **Inheritance 姿勢** —— 既存の複雑さへの制約感覚。洗練を通じて強化する。既存との共存を前提に置く
- **Emergent 姿勢** —— 常に別の方向性も探る。既存に引き戻されすぎず、**残り続ける構造はどれか**を問い直す

**Architect の出自は固定だが、姿勢は同じ人のなかで混ざる。** シニアな Architect ほど、両方の姿勢を状況に応じて使い分ける：

- バックエンドの Inheritance Architect でも、**前例のない機能**——新しいドメイン、新しい整合性要件、新しい運用モデル——を扱うとき、Inheritance 姿勢だけでは既存に引き戻されすぎて新しい骨格を立てられない。Emergent 姿勢を本人の中で立ち上げる必要がある
- フロントエンドの Emergent Architect でも、過去の蓄積を無視して新規性だけを追えば、チームに着地しない提案を量産する。Inheritance 姿勢の制約感覚が欠かせない

出自は「BE→Inheritance / FE→Emergent」というラフな領域マッチングが成立するが、**実行中の姿勢は両方を混ぜるのが標準**だと思っておいた方がいい。

### Anchor

**長命で品質の高いコードを置き、他者が書き換えにくい骨格を組む人。**

- シグナル：Quality ↑、Prod ○
- 現場像：
  - 「このスキーマ、誰が書いた？」「X さん」「**やっぱり。さすがだな**」——壊れない理由が先に腑に落ちる
  - コードレビューで、設計の筋を通すフィードバックを出す
  - 派手ではないが、触れた領域は組織の骨になる
  - Anchor が辞めると、その領域は数年にわたって重力を失う

Anchor は騒がない。目立たない。だから**観測なしでは見逃される**のが典型だ。Anchor が育った先にいるのが Inheritance Architect である。

### Cleaner

**他者の負債を清掃することで構造を守る人。**

- シグナル：Quality ↑、Survival ↑、Debt Cleanup ↑
- 現場像：
  - 他者のバグ修正を自分のタスクにする
  - リファクタリング PR を進んで出す
  - 新機能は少ないが、コードベースが彼の手を通った後は軽くなる

Cleaner は**組織のエントロピーを下げる装置**だ。新機能を作らないため成果物視点では評価されにくいが、Cleaner を失った組織は半年で目に見えて腐る。

### Producer

**量と速度で前に出す人。短期の推進力を担う。**

- シグナル：Prod ○
- 現場像：
  - スプリントのコミット数が多い
  - ローンチ直前に最も重宝される
  - 「とりあえず動くものを出す」が得意
  - 半年後に残っているかは、本人も気にしていない

Producer は**組織の速度**を作る。市場投入の締切が決まっている場面では Anchor より Producer が必要なフェーズがある。Producer が回すことで、**構造議論の対象となる動くコード**が先に生まれる。

> **AI 時代の注意**：Producer の「量と速度」は、**AI が最も代替可能な領域**でもある。コードを生み出す速度で差別化する時代は、既に終わりに近づいている。Producer 型に留まり続ければ、AI に置き換えられる確率が高い。High-Gravity Producer は Emergent Architect への進化経路を持つ——**進化なくして Producer の価値は薄まっていく**、という組織的前提を持っておく必要がある。

### Specialist

**一領域に深く、他には広がらない人。**

- シグナル：Survival ↑、Breadth ↓
- 現場像：
  - 自分の領域では誰より詳しい
  - 他の領域への関心は薄い
  - 異動・広報には向かない

Specialist は特定領域の Anchor だ。データベース設計、セキュリティ、決済基盤、インフラ自動化——こうした**技術的に複雑な領域**で価値を発揮する。

注意：Specialist は Breadth の低さで**過小評価されがち**だ。「リポ横断性が低い = 貢献が少ない」と短絡しないこと。狭くていい、深ければ。

## Style —— どう貢献しているか

Style は、Role と**独立に観測される**軸だ。同じ Anchor でも、Builder 風の Anchor と Mass 風の Anchor では見え方が違う。Style は「その人が仕事にどう向かっているかの手触り」を捉える。

| Style | シグナル | 要旨 |
|---|---|---|
| **Builder** | Prod ↑ + Design ↑ + Debt ○ | 積極的に設計を積み上げる |
| **Resilient** | Prod ↑ + Surv ↓ + RobustSurv ○ | 前進しつつも時間減衰に強く残る |
| **Rescue** | Prod ↑ + Surv ↓ + Debt ↑ | 負債清掃を引き受ける救援型 |
| **Churn** | Prod-Surv gap ≥ 30 + Qual ↓ + Surv ↓ | 書いたそばから書き換えられる |
| **Mass** | Prod ↑ + Surv ↓ | 量を出しているが残らない |
| **Emergent** | Gravity ↑ + Prod ○ + RobustSurv ↓ | 新しい構造的重力を生んでいる最中（収束前） |
| **Balanced** | Impact ≥ 30 | 全軸中庸 |
| **Spread** | Breadth ↑ + Prod ↓ + Surv ↓ + Design ↓ | 広く浅く、重心に触れない |

Emergent Style は、Role 節で触れた **Emergent Architect 系譜（High-Gravity Producer から進化する）が、まさに途中にある状態**を観測した姿だ。影響範囲と所有量は広がっているが、書いたコードはまだチームの衝突や書き換えに晒されていて Robust Survival は低い——これが「収束前の創発」として現れる。

Style は**否定的なラベルではない**。Mass も Churn も Spread も、それぞれ「いまそうなっている」という観測結果にすぎない。問題は Style それ自体ではなく、**Role との噛み合わせ**だ。

いくつか個別に補足する。

### Mass —— 量は出ているが残らない

- 毎日忙しそうに見える。コミットは多いが、3 ヶ月経つとほとんど書き換わっている
- 本人の疲労感と、組織への寄与が釣り合っていないように見える

Mass は、しばしば「**その組織・その配置**ではその Style に落ち着いてしまっている」というシグナルだ。同じ人が別の組織やレイヤーでは Builder / Resilient として機能することもある。Mass が長く続くのは、本人の責任というより**配置の失敗・役割設計の失敗**であることが多い。

複数人が Mass であれば、**組織が Mass を量産する構造**になっている可能性を示唆する——Producer 専任チームに陥っている等。

### Churn —— 書くたびに書き換えが必要になる

Churn は Mass と似て見えるが違う。Mass は「量を出しているが残らない」。Churn は「**書くたびに書き換えが必要になる**」。Churn の書き手は、しばしば意図せず他者の仕事を増やしている。

介入は慎重に。本人は一生懸命働いているが、コードベースに対しては純粋に**負の影響**を及ぼしている可能性がある。ペアプロと Anchor への弟子入りが効く。Churn は**訓練で変わる Style**であることが多い。

### Spread —— 広く浅く、重心に触れない

Spread の人は、真面目で活発だ。会議にも出て、多くのリポに触れる。しかし半年経ってコードを数えると、痕跡は薄い。

これは本人の能力不足ではなく、**Role と Style のミスマッチ**で起きる。Role が Architect / Anchor なのに Style が Spread になっている場合、配置で矯正できる可能性が高い。

#### 戦略的 Spread —— team-accelerating scaffolding

シグナル上は Spread に見えても、**それを戦略的に演じている**エンジニアがいる。

複雑な要件や、言語化が難しいプロダクト像を、**会話だけでチームに伝え切るのは無理**な局面がある。設計意図を説明しきる前に、実装の手が止まる。そこで、ある種のシニアは、自分で**大枠だけ動くコードをざっと書き切る**。細部の品質は意図的に追わず、後続に託す。

区別のヒントは、**本人が後続にバトンを渡せているか**、そして**バトンを受けた人の Survival / Design が向上しているか**。チーム全体の Robust Survival が、その人の関わったモジュールで上がっているなら、それは**意図された Spread**（team-accelerating scaffolding）だ。

「Spread だから配置換え」と短絡すると、**組織のドライブエンジンを壊す**ことがある。Role（Emergent Architect 候補など）を併読し、本人ヒアリングと併せて判断する。

## State —— ライフサイクルフェーズ

State は、Role / Style に重ねる**時間軸**の情報だ。その人が今どの段階にいるかを示す。

| State | シグナル | 要旨 |
|---|---|---|
| **Former** | RawSurv ↑ + Surv ↓ + (Design ↑ or Indisp ↑) | 過去に骨格を置いたが、今は離れている |
| **Silent** | Prod ↓ + Surv ↓ + Debt ↓（commits ≥ 100 のみ） | 静かに停滞している |
| **Fragile** | Surv ↑ + Prod ↓ + Qual < 70 | 変更圧がないから残っているだけ |
| **Growing** | Prod ↓ + Qual ↑ | 速度より質を上げる方向で成長中 |
| **Active** | 直近コミットあり | 現在も活動中 |

### Fragile の重要性

Fragile はとくに見落としやすい State だ。Survival が高いので**Anchor に見える**。しかし実態は、**触りたくても触れない**状態で凍結したコードの書き手である。

観測では Survival を **Raw Survival** と **Robust Survival** に分解し、**変更圧**を併読することで、Anchor と Fragile を切り分ける。Fragile は Raw ↑ / Robust ↓ かつ変更圧 ↓ のパターンで現れる。

Fragile が多い組織は、**変更不可能な負債**を抱え込んでいる。負債の書き手は残す意図がなくても「残ってしまった」だけで Anchor として扱われてしまう。観測でこの区別をつけないと、人事判断も配置判断も歪む。

### Former の扱い

Former は「過去の貢献」が残っている State だ。退職・異動した人の設計が今も効いているなら、Former として観測される。組織はしばしば Former の貢献を忘却するが、構造観測があれば**辞めた人への正当な敬意**を数字で保ち続けられる。

## 「能力はあるのに空転する人」の構造

組織にはしばしば、**能力は明らかに高いのに、組織の中で空転している人**がいる。本人も周囲も、何が起きているか言語化できない。

3 軸で読むと、これは特定の組み合わせとして現れる：

- **Role 強 × Style 弱** —— 本来 Anchor / Inheritance Architect の素養がある人が、緊急対応や会議調整で時間を溶かされて Style が Spread に落ちる。「この人、本来は深く座れるはずなのに、毎日の走り回りで薄くなっている」という状態。配置か仕事の切り出しの失敗
- **姿勢が単一で固まっている** —— BE で主要基盤を設計してきた Inheritance Architect が、会社として初の toC Native App 領域を任される。BE では出自と領域がきれいに合致して長年うまくいってきた。ところが新領域でも同じ Inheritance 姿勢のまま、既存の抽象をそのまま持ち込もうとする。FE / モバイルチームからのフィードバックが通らず、書き換えが続く。**問題は出自ではなく、姿勢が単一で固まっていること**。前節で述べた「姿勢は混ぜるのが標準」が、過去の成功体験に阻まれて実行できない
- **State の誤認** —— 前四半期と比べてコミット数は減っているが、Quality は着実に上がっている Architect 候補。本人は Growing State に入っているのに、「コミット少ない = 停滞 / Fragile 気味」と誤読されて評価が下がる

いずれも、**人の努力量を変えさせる介入**では解けない。**配置を変える**・**領域を変える**・**Style を訓練する**——構造側の介入で解ける。

## 型は固定ではない

繰り返し強調したいのは、これらの型が**その人の本質ではない**ということだ。

同じ人物が：

- 組織 A では Architect / Builder / Active
- 組織 B では Producer / Mass / Active
- 組織 C では Anchor / Spread / Growing

これは、本人が変わったのではなく、**組織のレイヤー構造と配置が違った**から起きている。

型は**組織と人の相互作用の観測結果**だ。だから、介入の対象は人ではなく、人と組織の接続部になる。これが第 4 章以降で展開される**レイヤー構造と配置設計**の話に繋がる。

## 現場で何が変わるか

人材を 3 軸で捉えると、組織の議論が変わる。

1. **採用計画が具体化する。** 「シニアを採る」から「Architect（Emergent 系）が 1 人足りない」「Cleaner が欲しい」に変わる
2. **評価の主語が変わる。** 「人の優劣」から「Role / Style / State の 3 軸で、今このフェーズに必要な構成が揃っているか」に変わる
3. **配置変更の議論が可能になる。** 「この人は Anchor / Mass / Active。配置を変えれば Anchor / Builder に化けるかもしれない」という会話が可能になる
4. **AI 時代への適応：** Producer の「量と速度」が AI で代替可能になるとき、組織に残すべきは Architect / Anchor / Cleaner の Role だ。**Producer 型に留まり続ける人の進化経路**を組織は設計しなければならない
5. **退職リスクの読みが変わる。** Inheritance Architect が辞めると、Anchor が育つ土壌が失われる。Emergent Architect が辞めると、新しい構造提案が止まる。失うものの質が事前に言語化できる
6. **自己認識が構造化する。** 本人が「自分は今 Producer / Mass 寄り。中期的には Emergent Architect に寄せたい」と考えられるようになる。キャリアが自己管理可能な対象になる

## 次章への接続

ここまで、人を 3 軸で捉える言語を整えた。しかし 3 軸だけでは組織は動かない。**型 × レイヤー**で初めて機能する。

同じ Architect でも、実装層に配置されている Architect と、中間層に配置されている Architect では、組織への影響が違う。同じ Producer でも、原理層に置かれたら空転する。

次章では、組織のレイヤー構造を定義する。実装層 / 中間層 / 原理層の三層と、それを繋ぐ**翻訳者**の役割。ここから、本章の 3 軸トポロジーが「配置設計」として実務に接続する。
