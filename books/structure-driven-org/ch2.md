---

> **この章のスコープ（骨子）**
>
> コードを「成果物」ではなく「構造」として読む。Design / Robust / Survival の意味を与える。

---

## 1. 「成果物」としてのコードの限界

- 成果物視点：行数、機能数、PR数 → **書かれた瞬間の量**しか見えない
- 結果、「書いた後に何が起きたか」が評価から抜け落ちる

## 2. 構造視点に切り替える

- コードは**書かれるもの**ではなく、**残るもの / 書き換えられるもの / 他者に継承されるもの**
- ここで 3 つの軸を導入：
  - **Design** — どの層のファイルに触れたか（アーキテクチャ的重心）
  - **Robust** — 時間減衰下でも生存しているか（`exp(-days/τ)`）
  - **Survival** — 他者による書き換えを経ても残存する比率

## 3. なぜコードは書き換えられるのか

- 設計の誤り？ 要件変化？ スタイルの不一致？
- 書き換えの**率**ではなく、書き換えの**勾配**を見ることで、構造の強度が見える

## 4. Git 考古学の具体手順（概念スケッチ）

1. `git log --numstat` で変更量と拡散を取る
2. `git blame` で現在時点の帰属を取る
3. 時間減衰を掛けて Robust Survival を算出
4. Design weight（アーキ重心ファイル）を掛ける

## 5. 現場で何が変わるか（予定）

- コードレビューの視点が「スタイル」から「この変更はどれくらい残る設計か」に変わる
- ローンチ直前の派手な貢献と、半年後に残る貢献を区別できる

## 6. 次章への接続

- 構造として読めるようになった**コード**から、それを生み出す**人**の構造モデルへ。
