*コードは書かれた瞬間ではなく、書かれた後に何が起きたかで測る。残ったものが、構造だ。*

---

> **本章の射程**：設計レイヤー（コードという成果物を「構造」として読み直す観測軸を定義する）

### こんな場面から始まる

新人 X が初めて出した PR は、教科書通りに綺麗だった。命名規則も完璧、テストも揃っていて、レビュアー全員が即座に LGTM を押した。同じ週、同期入社の Y が出した PR は、命名がややバラついて変数名も省略多めで、レビューで「あとでリファクタしましょう」のコメントが付いた。Y は素直に直すと言った。

半年後。X が書いたあの「綺麗な PR」のコードは、ほぼ全てリファクタリングされて消えていた。Y のコードは、レビューで指摘された箇所も含めて、ほぼ全て残っていた。「綺麗だった X」と「雑だと言われた Y」、構造への寄与で見れば順位が逆転していた。

本章は、「**書かれた瞬間の綺麗さ**」と「**書かれた後にどう生き残るか**」を区別する語彙を組む。

---

## コードは成果物か、構造か

「このエンジニアの成果物は何ですか」と問われて、多くの組織は答える：

- リリースした機能
- クローズした PR
- 書いたコード行数
- 解決したチケット

いずれも、**書かれた瞬間に存在した量**だ。時間軸に対して、**ゼロから正への移行**しか見ていない。

しかしコードは、書かれた後にも動き続ける。書き換えられる、他者に引き継がれる、捨てられる、機能する、壊れる。書かれた瞬間の量だけでは、**コードが書かれた後の人生**がまったく見えない。

本書はここで視点を切り替える。**コードは成果物ではなく構造である**。構造として読むとは、時間軸の上で何が残り、何が消え、何が書き換えられたかを追うということだ。

## 成果物視点の限界

成果物視点でコードを扱うと、いくつかの具体的な歪みが生まれる。

### 1. 短命コードと長命コードが区別できない

1000 行のコードを書いた A と、100 行のコードを書いた B。従来の指標では A の方が 10 倍の貢献だ。

しかし半年後、A のコードはほぼ全て書き換えられ、B のコードは全て残っているとする。書き換えは無駄ではない——次の開発者がその上で試行錯誤する足場にはなった。ただし **残ったコード**という観点で見れば、B の 100 行の方が**構造への寄与は大きい**。

成果物視点は、この区別をできない。結果、**書いた瞬間にだけ注目される職場**では、短命コードを量産する戦略が合理的になる。

### 2. リファクタリングとクリーンアップが評価されない

過去の誰かが書いた負債を清掃する作業は、新機能を作らない。コード行数は増えないか、むしろ減る。チケット的には地味だ。

しかし、**これをやる人がいないと組織は腐る**。コードベースは、人が意図して清掃しなければ、時間とともにエントロピーが増大する一方だ。

成果物視点は、この種の貢献を見落とす。清掃する人は「目立たないが必要な人」として、恩恵を受ける側から曖昧に感謝されるに留まる。構造視点なら、**清掃による負債減少量**を観測できる。

### 3. 書き換えの勾配が見えない

同じ「書き換え」でも、意味は違う。

- 良い設計の上で、次の世代がそれを継承しながら書き換える
- 悪い設計に気づいて、次の世代が書き直す
- 書き手と引き継ぎ手の意図がずれて、繰り返し書き換わる

この 3 つは、書き換え率という単一の数字ではまとめられない。**書き換えがどう起きているか**を構造的に見る必要がある。

成果物視点は、「書き換えが起きた」という事実しか捉えない。構造視点は、そこから**次の判断に使える情報**を引き出せる。

## コードを構造として読む 3 軸

本書では、コードを構造として読むために 3 つの軸を使う——**Design（設計）**、**Survival（生存）**、**Change Pressure（変更圧）**。Survival はさらに 2 つの測り方に分かれる。

### Design（設計）

**どの層のファイルに、どれだけ寄与したか。**

組織のコードベースには、重心がある。エントリポイント、認証基盤、ドメインモデル、データスキーマ、共通ライブラリ——これらは他の多くのコードから依存されている。ここに触れる変更は、**設計の重心を動かす**変更だ。

Design 軸は、「どのファイルに触れたか」で重み付けされる。周辺的なユーティリティファイルを 1000 行触っても重心は動かない。核心的な 1 ファイルを 10 行触ったら重心が動く。

EIS は、アーキテクチャ重心ファイルのリストを設定で受け取り、そこへの貢献を加重する。

### Survival（生存）

**書いたコードがどれだけ残っているか。** ここには 2 つの測り方があり、それぞれ意味が違う。

#### Raw Survival

**現時点で、他者の書き換えを経ても残っている比率。**

Raw Survival は最もシンプルだ。`git blame` で現時点の帰属を取り、当初の書き手ごとに残存量を集計する。半年前に書いた 100 行のうち、30 行が現在も自分の blame で残っていれば、Raw Survival は 30%。

#### Robust Survival

**時間減衰で重み付けした生存量。**

Raw Survival は「現在も残っている行数」を素朴に数える。これだけでは、5 年前に書かれた行と先週書かれた行が、どちらも同じ 1 行として扱われてしまう。結果として、**長く在籍しているほど累積で値が大きくなる**——つまり時間でゲームできてしまう。

そこで Robust Survival は、生存している各行に**書かれてからの経過日数に応じた重み**を掛けて合算する。EIS では `exp(-days / τ)` の形を取る（`days` は経過日数、`τ` は減衰定数）。**古い行ほど重みが軽くなる**設計だ。

この設計意図は 2 つある：

1. **ゲーム耐性**：在籍年数でスコアを積み上げられないようにする。古い行の寄与は時間とともに減る
2. **現在への重み付け**：コードベースは時間とともに進化する。**現在も活きている直近のコード**を相対的に重く扱うことで、今の構造への寄与が見えやすくなる

#### Raw と Robust をなぜ分けるのか

Raw Survival だけでは、「**古くから残っているコード**」と「**最近書かれて今も残っているコード**」を区別できない。Robust Survival を並べることで、生存量の**時間的な内訳**が初めて読める。

ただし「古くから残っている = 構造に貢献している」とは限らない。**誰も触れないから残っているだけ**の可能性もある。Raw と Robust を並べても、この区別までは完全には切り分けられない。最終的にそれを決めるのは、次の軸——変更圧だ。

### Change Pressure（変更圧）

**ある領域に、どれくらい変更が集中しているか。**

定義は単純だ：

```
change_pressure = commits_touching_module / module_LOC
```

ある期間にモジュールに触れたコミット数を、モジュールのサイズ（LOC）で割る。大きなモジュールに多くのコミットがあっても圧は並みで、小さなモジュールに同じコミット数なら圧は高い。

変更圧が重要になるのは、**Survival の解釈を反転させうるから**だ。

- **変更圧が高い**領域で Survival が高い → そのコードは**変更の圧を受けながら耐えている**。本物の骨格。
- **変更圧が低い**領域で Survival が高い → そのコードは**耐えたのではなく、誰も触れないから残っている**。凍結した負債の可能性。

Survival だけで人を評価すると、後者の書き手まで骨格とみなしてしまう。変更圧を重ねることで、**Robust な骨格**と**触れない凍結地帯**を見分けられる。

### 3 軸の関係

3 軸は独立ではない。互いに情報を補い合う。

- Design ↑ + Survival ↓ → 重心に触れたが定着させられなかった
- Design ↓ + Survival ↑ → 周辺ファイルで長命コードを書いた
- Design ↑ + Robust Survival ↑ + **変更圧 ↑** → **本物の Anchor**（圧を受けながら骨格が残っている）
- Design ↑ + Raw Survival ↑ + **変更圧 ↓** → **Fragile な骨格**（触られないから残っているだけ）

一軸だけで評価すると必ず歪む。**3 軸を同時に見ることで、初めて構造が言語化できる**。

> 変更圧は、EIS を「単なるインパクト指標」から「**engineering risk detector**」へ進化させる核になる観測軸だ。本書では 3 軸のひとつとして扱うが、リスク検知の文脈では独立した主役になる。

## なぜコードは書き換えられるのか

構造として読むとき、「書き換え」は敵ではなく情報源だ。書き換えの**率ではなく勾配**を見る、という話を序論から持ち込む。

書き換えの原因は、大きく分けて 4 つある。

### 1. 設計上の誤り

最初の設計が、後の要件に耐えられなかった。これは誰にでも起きる。完璧な設計は存在しない。問題は、「設計の誤りが書き換わったコード」と「設計の正しさが時間に耐えたコード」を**区別できるか**だ。

Robust Survival を見ると、この区別ができる。短期に大きく書き換わった部分は、設計の誤りによる書き換え。長期にわたって徐々に書き換わっていく部分は、自然な進化。

### 2. 要件の変化

ビジネス要件が変わって、コードを追随させた。これは設計の誤りではない。要件変化に追随できたこと自体が**設計の柔軟性**の証拠だ。

Design 層の変更が、**要件変化のたびに適切に波及する**組織は、健全な構造を持っている。Design 層だけ孤立していて、要件が変わっても反映されない組織は、構造が腐っている。

### 3. スタイルの不一致

書き手と引き継ぎ手の好みが違って書き換えた。これは意味のない書き換えに近い。少数なら問題ないが、**これが主要な書き換え原因になっている組織は、コードベースが政治的**だ。

スタイル統一のチャンスは、ペアプロと継続的なレビューで取る。書き換えで取るのは遅い。

### 4. 新しい理解

書いたあとで、「こう書けばよかった」と気づいた。リファクタリングだ。これは必要悪というより、**構造を育てる必然的プロセス**だ。

この種の書き換えは、Debt Cleanup シグナルとして観測できる。他者の過去の負債を、今の理解で清掃する量。これが多い人が、組織のコードベースを健全に保っている。

書き換えを 1 つの数字に潰さず、**勾配で見る**と、これらの区別が可能になる。

## Git考古学の具体手順（概念スケッチ）

Design / Survival / 変更圧 を、実際の git リポからどう算出するか。概念レベルで手順を示す。

```
1. git log --numstat --all                 # 全コミットの変更量
   → コミット粒度の (author, file, +/-) を取得

2. git blame <file> at HEAD                # 各ファイルの現時点の帰属
   → Raw Survival の算出
     （現在残存している行が、誰によるものか）

3. 時間減衰係数 exp(-days/τ) を各行に適用
   → Robust Survival の算出
     （Raw Survival を時間減衰で重み付けしたもの）

4. Design weight: アーキテクチャ重心ファイルリストで
   各コミットを重み付け
   → Design 貢献の算出

5. Change Pressure:
   module 単位で
   commits_touching_module / module_LOC を算出
   → 変更圧の算出
     （Survival が「耐えた」か「凍結」かを切り分ける）

6. Cleanup detection:
   直前のコミット（他者）を書き換えているコミットを検出
   → Debt Cleanup の算出

7. Ownership map:
   モジュールごとに blame 比率を集計
   → Indispensability（バスファクター）の算出
```

これらをすべて機械的に算出するのが EIS だ。1 リポにかかる時間は数秒〜数分。組織横断で回しても半日かからない。

**観測に時間をかけない**のが重要だ。観測が重い装置だと、運用に乗らない。EIS は設計上、**毎週あるいは毎月定期実行**できる軽さを持っている。

## 具体例：同じ人物が、別のコードベースで違う顔を見せる

前章の A・B・C の例を、別のコードベースに移して考えてみる。

B（Anchor タイプ、Survival 高）が、新しい若手主導のスタートアップに入ったとする。このコードベースはまだ 2 年しか経っておらず、すべてのコードが書き直しの途中だ。Survival を測ってもノイズしか出ない。

ここで B は、従来の定義では「Anchor」として機能しない。しかし、**この組織の Design 層は誰も触れていない**——スキーマ設計、アーキテクチャ方針、認証基盤など、骨格にあたる部分が未整備だった。

B がそこに着手して 3 ヶ月経ったとき、Design 軸が急激に立ち上がる。Survival はまだ出ないが、**Design に寄与した履歴**が残る。さらに 6 ヶ月経つと、B が書いた Design 層のコードが、他者によって継承される形で書き換えられ始める。Survival が徐々に立ち上がる。

これが「**構造的に読むと、同じ人物の貢献が組織によって違う顔を見せる**」ということだ。A の頑張り・B の頑張り、といった絶対的な比較はできない。しかし**どの組織のどのフェーズで、どの軸に寄与しているか**は、構造観測で見える。

## 現場で何が変わるか

構造としてコードを読めるようになると、以下が変わる。

1. **コードレビューの語彙が変わる。** 「このコードは綺麗」ではなく「このコードは Design 層に寄与している / 時間に耐えて Robust になりそう / 変更圧の高い領域で書かれている」と言える
2. **ローンチ直前の派手な貢献と、半年後に残る貢献が区別される。** 短期の成果と長期の成果が別の指標で可視化される
3. **リファクタリングが初めて評価対象になる。** Debt Cleanup 軸があれば、「今期は清掃で 30% の負債を減らした」と言語化できる
4. **新規プロジェクトで誰を呼ぶかの判断が変わる。** Anchor が欲しいのか、Cleaner が欲しいのか、Producer が欲しいのか、軸で判断できる
5. **退職者が残したものの評価が変わる。** 「辞めた人が書いたコードがどれだけ残っているか」で、その人の組織への恩恵を事後評価できる

5 番目は特に重要だ。従来の組織は、**辞めた人の貢献を忘れる仕組み**になっている。構造観測があれば、辞めた人が組織の Design 層に残したものを、数字で見続けられる。これはその人への正当な敬意でもある。

## 次章への接続

ここまでで、コードを構造として読む軸を用意した。次に問うべきは、**その軸で見えるコードを、誰が書いているか**だ。

人を「能力」で見ると、外からは観測できない。人を「構造への寄与の型」で見ると、観測できる。同じ人が、全く違う軸の配分で構造に寄与していることが見えてくる。

次章では、Anchor / Producer / Mass という 3 つの基本型と、そこから派生する型を導入する。これは性格タイプ分類ではなく、**構造観測の結果としてのパターン**だ。その前提で読み進めてほしい。
