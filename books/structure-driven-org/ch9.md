*組織論は、書籍かコンサルで閉じるのが 100 年の慣習だった。そこに OS を作る選択肢が加わったのは、観測が成り立ったからだ。*

*この章だけは、理論の本から一歩外に出る。扱うのは、構造駆動エンジニアリング組織論が**プロダクトとして立ち上がる瞬間**だ。*

---

> **本章の射程**：実装レイヤー（EIS CLI と OrbitLens Ace SaaS の分業、組織 OS としての Ace の設計方針）+ 戦略レイヤー（組織論が SaaS になる瞬間の論理化）。CLI は **EIS**、SaaS は **OrbitLens Ace** と本章で呼称を固定する。

### こんな場面から始まる

**シーン A（人力観測の限界）**

120 人のエンジニアリング組織、リポジトリは 30 近く分散している。EM はクォーター頭に各リポで EIS を走らせ、JSON を手元に集めて、Design 中央値と Fragile モジュールの分布を可視化した。翌週、ある機能チームで事故が起きる。「最新の数字を取りたい」と EIS を再実行する——**CLI 自体はキャッシュが効いて数分で返る**。しかし、複数リポの結果を突合し、前回からの差分を作り、Slack とダッシュボードに流すのは**全て手作業**。対応が終わってから、先月の観測をもとに事後分析する。**観測が意思決定に間に合わない**。3 ヶ月後、EM は観測を手放す——**CLI が遅いのではなく、継続運用の配管コストが意思決定の便益を上回る**からだ。

**シーン B（組織が OS として動いている）**

同じ組織に OrbitLens Ace が常駐している。PR マージのたびに EIS シグナルが再計算され、Slack に構造ダッシュボードの差分が流れる。Fragile になり始めたモジュールは、所有者本人に先に通知が届く。1on1 を開くと、構造コンテキスト——相手のアーキタイプ、触っているモジュールの Vitality、直近 4 週間の Survival 推移——が自動で会議テンプレに埋め込まれている。**観測が組織の配管に溶け込んでいる**。EM は観測を回す人ではなく、観測の上で**判断する人**に戻る。

---

## 1. 観測を人力で回す限界

EIS は単発実行では十分に強力だ。Git だけで 7 軸のシグナルを出し、アーキタイプとモジュールトポロジーを JSON で吐く。CLI でローカル分析する限りは、何の足りなさもない。

しかし **継続観測** になった瞬間、運用コストが指数的に上がる。観測の対象が **人 × モジュール × 時間** の 3 次元になるからだ。

- 人が 100 人を超えたら、個別レポートを毎週人力で回すのは不可能
- モジュールが 200 超になれば、Fragile 化の兆候を目視で追えない
- 時間軸を保持しない観測は、**「いまの組織の健全性」は語れても「先月から今への推移」は語れない**

ここで、観測を組織の外に常駐させる装置——**組織 OS としての SaaS**——が要る。

## 2. 望遠鏡と組織 OS

CLI と SaaS の責務分離は、シンプルに書ける：

- **EIS（CLI）＝ 望遠鏡** —— Git という生データから観測シグナルを生成する道具
- **OrbitLens Ace（SaaS）＝ 組織 OS** —— 観測シグナルを受け、組織が動くための配管全体を提供する基盤

初期の設計では、SaaS を「**天文台**」と呼んでいた。観測データを集めて解釈する装置、という比喩だ。しかし書き進めるうちに、**これは射程が狭すぎる**と気付いた——次節で詳述するが、組織は観測データを眺めて終わりではなく、**観測を受けて動く**。観測 → 解釈 → 介入 → 再観測、このループ全体を配線しているものを、組織 OS と呼ぶ。

そこで Ace の位置付けは「天文台」から「組織 OS」へ広がる。観測解釈だけでなく、**介入の配管、記録、次の観測サイクルへのフィードバック**までが射程に入る。

CLI に推薦・予測・介入テンプレは入れない。EIS は Git から読めるものだけを読む単機能ツールとして閉じる。**CLI は望遠鏡、SaaS は組織 OS**——役割の分離は本書の一貫ルールだ。

> **呼称の整理**：CLI の名前は **EIS**。SaaS の名前は **OrbitLens Ace**。どちらも単独で「ace」と呼ばない——EIS の発音が「ace」なのと SaaS のプロダクト名末尾が「Ace」なのが被るため、単独使用は曖昧になるから。

## 3. Ace の機能領域 — 観測 SaaS から組織 OS へ

Ace の機能領域は、**観測解釈 → 介入配管 → 再観測ループ** の 3 層で設計されている。

**観測解釈層（現在の主戦場）**

- **Structural Summary** —— 組織の構造サマリ。どのアーキタイプが何人、どのモジュールが Stable / Fragile / Turbulent / Critical / Dead
- **人 × モジュール突合** —— Conway's Law の検証。組織構造とモジュール境界の整合・不整合を検出
- **時間系列リスク予測** —— Fragile Fortress の早期検知。コミット変動と tested/untested survival の推移から兆候を読む
- **アラート** —— 観測シグナルの変化率に応じた通知

**介入配管層（組織 OS 化の要）**

- **介入テンプレート** —— 1on1・レビュー・ペアプロ・組織変更を、第 6 章の 3 層（行動 / 出力 / 蓄積）で分解したフォーマット
- **介入記録** —— 誰が誰に、どの介入を、どの文脈で行ったか。対応する観測データへの遡及リンク
- **構造語彙ダッシュボード** —— チーム内での構造語彙流通度の時系列表示（第 7 章の文化シグナル）
- **文化シグナル統合** —— 会議ログ、コード・PR、Git 考古学の 3 バケツを統合して表示

**再観測ループ層**

- 介入実施後、**次の観測サイクルで効果を検証**する配管
- 介入 → 観測 → 解釈 → 次の介入、のループが Ace 内で閉じる

> **現在の開発状況**
>
> OrbitLens Ace は**現在開発中**だ。観測解釈層を最初のフェーズとして出荷に向けて作っている段階で、介入配管層と再観測ループ層は段階的に追加していく。本章で描くのは Ace の**到達目標としての組織 OS 像**であり、今のプロダクトはその途上にある。「書籍として構造駆動論を固定し、プロダクトとして段階的に配線していく」——この並走自体が、構造駆動を**書籍で閉じない**ための設計だ。

## 4. ブランドとしての OrbitLens

![OrbitLens](https://raw.githubusercontent.com/machuz/eis/main/docs/images/blog/sdo-ch9-orbitlens-inversion.svg)

- 外部表記：**OrbitLens**（本書・LP・外部ドキュメント）
- 内部表記：**orbitlens**（リポジトリ・コード・社内ドキュメント）
- 法人登記：**Orbitlens, Inc.**

三層プロダクト構成：

- **OrbitLens Ace** —— 組織 OS SaaS 本体。EIS 観測を解釈し、介入の配管・記録・再観測を担う
- **OrbitLens True** —— 登録導線。クロスボーダーでのエンジニア × 組織マッチング
- **OrbitLens Ideal** —— 将来枠。構造駆動エンジニアリング組織論の運用層として育てる領域

3 つは独立したプロダクトだが、**同じ構造語彙の上に立っている**——7 軸・アーキタイプ・3 層・変換・介入 3 層・文化シグナル。共有語彙が、プロダクト間を配管する。

## 5. 組織論が SaaS になる瞬間

この本を書き始めたとき、OrbitLens Ace は **観測データの解釈ツール** として構想していた。CLI が吐く JSON を取り込み、可視化とアラートを返す、程度の設計だ。天文台の喩えがそこから来ている。

書き進めるうちに、**この設計が狭すぎる**ことに気付いた。

構造駆動エンジニアリング組織論は、観測で閉じない。第 6 章で扱った介入設計（行動 / 出力 / 蓄積）、第 7 章で扱った文化の観測シグナル、第 8 章で扱った再観測ループ——これらすべてが**観測を前提にした配管**であり、**同じ語彙の上で動く**。観測だけを SaaS 化して、介入配管を手作業に戻すのは、**中途半端**だ。

この気付きから、Ace の射程が広がった。観測 → 解釈 → 介入 → 再観測のループごと、一つの SaaS で回す。組織論全体が、同じ配管の上に載る。**組織 OS としての SaaS**、という位置付けが定まった瞬間だ。

組織論は、**書籍**か**コンサル**で閉じるのが 100 年の慣習だった。理論は読めるが運用はコンサルタントの頭にあり、会社を変えるたびに運用ノウハウは失われた。構造駆動は**観測を土台に設計されている**ので、配管ごと SaaS にできる。**これが、組織論がプロダクトになる歴史的瞬間**だ。

## 6. 境界線 — SaaS は何をしないか

組織 OS として広げても、Ace が**やらないこと**は明示的に残す。

- **介入そのものは、現場の人間の仕事**。Ace は介入の材料を出し、記録し、次の観測に繋ぐところまで。「人間が人間に声をかける」瞬間を SaaS が代替することはない
- **信頼の蓄積、非言語の了解、人間関係の機微**は観測しない（第 8 章の観測倫理に従う）
- **評価・懲戒への直接接続は構造上禁止**。Ace のダッシュボードは**組織の自己修正のための材料**であり、個人の査定資料ではない。混ぜた瞬間、現場は自然な発話を止めて文化が死ぬ
- **代替的な観測パス**は意図的に複数残す。Slack、会議、1on1、レビュー——人間の目がアクセスできる経路を潰さない

観測 → 解釈 → 介入 → 再観測 の 4 段のうち、Ace が担うのは**観測・解釈・記録・次観測**の 4 段。**介入を人間が行う** という 1 段が、意図的に残される。人間の判断を SaaS に預けないための境界線だ。

## 7. 現場で何が変わるか

構造駆動組織が Ace の上に載ると、以下が変わる：

1. **観測の運用負荷が消える** —— PR マージごとに EIS が再計算され、ダッシュボードが更新される。EM が手元で CSV を捏ねる世界は終わる
2. **組織の変化が時系列で残る** —— これまでの組織論は「現時点の組織の健全性」は語れても「3 ヶ月前から今への推移」は語れなかった。Ace は時間軸で変化を保持する
3. **介入が属人化から解放される** —— 1on1 のテンプレ、レビューの語彙辞書、組織変更のチェックリストが SaaS 内で管理される。**辞めた人の頭の中にあった運用が、組織に残る**
4. **文化シグナルが可視化される** —— 第 7 章で定義した 3 バケツが統合ダッシュボードとして常駐する。文化の定着度が、数字で読める
5. **経営と現場が同じ言語で組織を議論できる** —— 取締役会でも全社会議でも、同じ構造シグナルと介入設計の 3 層で話せる

## 8. 次章への接続

Ace が組織 OS として組織に常駐する世界——そこで問われるのは「組織を管理するとは何か」ではなく、**「OS を作るとは何か」**だ。

次章（最終章）は、構造駆動エンジニアリング組織論が最終的に何を作っていたかの総括。**管理でも文化でもない、組織の OS を作る**、の意味を閉じる。
