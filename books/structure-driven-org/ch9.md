*組織論は、書籍かコンサルで閉じるのが 100 年の慣習だった。そこに OS を作る選択肢が加わったのは、観測が成り立ったからだ。*

*この章だけは、理論の本から一歩外に出る。扱うのは、構造駆動エンジニアリング組織論が**プロダクトとして立ち上がる瞬間**だ。*

*先に断っておく——このプロダクト群は、評価を自動化するためのものではない。観測と自己修正の配管を、組織に残すためのものだ。*

---

> **本章の射程**：実装レイヤー（CLI = **EIS** / 観測 SaaS = **OrbitLens Ace** / 組織 OS SaaS = **OrbitLens Ideal** の役割分担）+ 戦略レイヤー（観測の軽さと組織 OS の重さを別プロダクトに分けて、構造駆動論全体をプロダクトとして立ち上げる設計）。呼称はこの章で固定する。

### こんな場面から始まる

**シーン A（人力観測の限界）**

120 人、リポジトリ 30。EM はクォーター頭に EIS を走らせ、Design 中央値と Fragile モジュールの分布を可視化した。翌週、機能チームで事故が起きる。EIS を再実行する——**CLI は数分で返る。組織の意思決定には、間に合わない**。複数リポの突合、差分生成、Slack とダッシュボードへの配信は全て手作業。事後分析の頃には、観測は先月のままだ。3 ヶ月後、EM は観測を手放す。**継続運用の配管を自前で組み続けるコスト**が、意思決定の便益を上回るからだ。

**シーン B（観測が手元に置いてある）**

同じ組織に OrbitLens Ace が導入されている。EM は朝イチで Ace を開く。Design の中央値、Fragile 化しかけのモジュール、アーキタイプ分布、直近 2 週間の推移が、1 画面に収まっている。昨日の PR マージで Survival がどう動いたかも読める。Slack には日次の差分サマリが流れている。特別な運用は要らない——**シグナルが、いつもそこにある**。EM は観測を回す人ではなく、観測を**読む人**に戻る。観測が軽く、気楽に、手元に置いてある状態。これが Ace が提供する体験だ。

## 1. 観測を人力で回す限界

EIS の単機能は十分に揃っている。`eis analyze --recursive` で複数リポ横断、`eis team` でチーム集約、`eis timeline --span 3m --periods 0` で時系列履歴、`eis analyze --author` で個人フィルタ——組織観測に必要な機能ブロックは、ほぼ CLI で取り出せる。CLI が「機能的に足りない」わけではない。

限界は別のところにある。**CLI の上に、組織運用の配管を自力で組み続けるコスト**だ。具体的には：

- **定期実行と履歴の永続化** —— 毎日・週次で自動実行し、結果を組織として保持する配管。cron + S3、GitHub Actions + DB、どこかで誰かが書く
- **ダッシュボード** —— 複数リポ・複数軸・時系列を重ねて一覧する UI。CLI の出力（JSON / CSV）を読み取って可視化する層を自分で作る
- **通知・Slack 統合** —— Fragile 化の兆候、Survival の急落、アーキタイプ分布の変動を、能動的に届けるフック
- **アクセス制御と可視化モード** —— 誰が何を見られるか、匿名化は要るか、個人合計を出すか——評価と観測を切り離すための制御層
- **経営と現場の共有画面** —— 同じ観測を、取締役会でも機能チームでも、同じ表現で読める共有層
- **運用ノウハウの組織保有** —— 配管を組んだ人が辞めた瞬間に観測が止まる構造を回避する

これらは CLI が「やらない」ものではなく、**CLI の外に組み立てる必要がある**もの。組織が 20 人なら手書きで回せる。50 人程度まではスクリプトで何とかなる。100 人・数十リポを超えると、**配管の保守自体が専任業務**になる。そして、作った人が辞めた瞬間、観測も止まる。

本書が語るのは思想と設計であって、配管の実装主体は問わない。OrbitLens を使わないチームは、上記 6 要素を自前で組み立ててもよい——その選択は正当だ。ただし、**配管の保守と継承のコストは、そのまま自組織のものとして残る**。

**観測を人力で回す限り、それは文化ではなく個人技で終わる**。観測を組織の外に常駐させる装置——**観測 SaaS**——が要る。

## 2. 望遠鏡、天文台、組織 OS

CLI と SaaS の役割を、**3 段の比喩**で整理する：

- **EIS（CLI）＝ 望遠鏡** —— Git という生データから観測シグナルを生成する道具
- **OrbitLens Ace（SaaS）＝ 天文台** —— 観測シグナルを readable な形に整え、**手元に置く**装置。気楽に使えることそれ自体が価値
- **OrbitLens Ideal（SaaS）＝ 組織 OS** —— 観測を前提に、介入の配管・記録・再観測ループを配線する基盤

なぜ 2 つの SaaS に分けるのか——**観測の軽さ**と**組織 OS の重さ**は、別種の UX だからだ。

- **観測は軽く、気楽に使えることが価値**。個人の EM が朝イチで開いて、今日どこを見るか決める。そのための道具は、**手元に置ける軽さ**と**即座の示唆**を最優先で設計する
- **組織 OS は重く、配管が価値**。1on1 テンプレ、介入記録、文化シグナル、再観測ループ——これらは組織全体の運用層として組み上げる必要があり、導入にも運用にも重みがある

この 2 つを 1 プロダクトに押し込むと、**Ace の軽さが死ぬ**か、**Ideal の深さが出ない**か、どちらかが起きる。**Ace と Ideal の分離は好みではない。構造上の必然である**——観測の UX と組織 OS の UX は、同じプロダクトの中で両立しない。分けない設計を選んだ瞬間、どちらかが壊れる。

CLI に推薦・予測・介入テンプレは入れない。EIS は Git から読めるものだけを読む単機能ツールとして閉じる。**望遠鏡 / 天文台 / 組織 OS**——役割の 3 段分離は本書の一貫ルールだ。

> **呼称の整理**：CLI の名前は **EIS**。観測 SaaS の名前は **OrbitLens Ace**。組織 OS SaaS の名前は **OrbitLens Ideal**。どれも単独で「ace」とも「ideal」とも呼ばない——EIS の発音が「ace」で SaaS 名末尾の「Ace」と被り、また「ideal」は形容詞として別用途の語なので、単独使用は曖昧になるから。

## 3. Ace の機能領域 — 観測を readable に、手元に

**Ace は管理ツールではない。組織の現在地と不足を読むための装置である**——観測を、人間の認知が扱える速度まで落とし、構造のギャップを発見させる。

Ace の機能は、**観測・解釈・ギャップ発見**に絞られる。組織 OS 的な配管は持たない。

- **Structural Summary** —— 組織の構造サマリ。どのアーキタイプが何人、どのモジュールが Stable / Fragile / Turbulent / Critical / Dead
- **人 × モジュール突合** —— Conway's Law の検証。組織構造とモジュール境界の整合・不整合を検出
- **時間系列リスク予測** —— Fragile Fortress の早期検知。コミット変動と tested/untested survival の推移から兆候を読む
- **アラート** —— 観測シグナルの変化率に応じた通知、Slack への日次差分配信

Ace の本質は「見える化」ではない。**解釈の加速・情報のブレイクダウン・参照性の向上・示唆の取り出しやすさ**——これらが揃うことで、観測データが意思決定の材料として機能する。生の JSON を吐くだけなら CLI で足りる。Ace の役割は、**人間の認知がデータに触れやすい形に観測を再整理し、手元に置く**ところにある。

**「気楽に使えることが価値」**——これが Ace の設計思想だ。重い運用フローも、分厚い導入プロセスも、管理者権限の分岐も、最小限に抑える。EM 個人でも、少人数チームでも、1 日目から開ける。Linear 的な軽さ、Raycast 的な即応性を、組織観測に持ち込む。

もう一つ、Ace の差別化を構成するのは、**True / Ideal への接続性**だ。Ace で組織の穴が見えた瞬間、その穴をどう埋めるかが同じ構造語彙の上で 2 方向に分岐する:

- **外から埋める → True への接続** —— 構造ギャップ（Design 担う Architect が不足、特定モジュールで Indispensability が偏っている）が、そのまま**採用／マッチング条件**になる。「Architect 相当の候補を探す」「この Indispensability 偏重を緩和できる候補」が、Ace のシグナルから直接 True の探索条件に落ちる
- **内で回す → Ideal への接続** —— Ace で読んだ観測が、Ideal 側の**配置提案・介入テンプレ・記録**に引き継がれる。「先週の 1on1 で扱った Fragile 化モジュールが、今週 Ace でどう変化したか」が、同じ画面で閉じる

単体ツールとしての観測 SaaS は他にもある。Ace の立ち位置は、**観測から人材、観測から運用、までが一つの語彙で貫通する**ことだ。観測だけ、採用だけ、運用だけを個別に足し算したプロダクト群に対して、構造駆動の**統一語彙**が接続コストをほぼゼロにする——ここが、OrbitLens 全体として取りに行くポジションだ。

## 4. Ideal の役割 — 組織 OS としての運用層

**Ideal は運用支援ツールではない。今いる人員で組織を回すための、組織 OS である**。

観測だけでは、組織は動かない。Ace で見えた現在地と不足を、**今いる人員（True 経由で新たに加わった人員も含む）で回す**配管を担うのが Ideal だ。観測を受けて、**配置し、介入し、記録し、再観測する**——この内部のループを組み上げる。

- **配置提案** —— Ace で見えた構造ギャップ（Architect 不足、Indispensability 偏重など）に対し、**社内の既存メンバーで誰をどこに配置換えするか**を提案する（内部充足）
- **介入テンプレート** —— 1on1・レビュー・ペアプロ・組織変更を、第 6 章の 3 層（行動 / 出力 / 蓄積）で分解したフォーマット
- **介入記録** —— 誰が誰に、どの介入を、どの文脈で行ったか。対応する観測データへの遡及リンク
- **構造語彙ダッシュボード** —— チーム内での構造語彙流通度の時系列表示（第 7 章の文化シグナル）
- **文化シグナル統合** —— 会議ログ、コード・PR、Git 考古学の 3 バケツを統合表示
- **再観測ループ** —— 介入実施後、次の観測サイクルで効果を検証する配管

Ideal は、**Ace の上に組み上げる**。Ace が手元に置いた観測を、組織運用の配管に接続する層、という位置付けになる。導入は重く、運用には組織としての合意が要る——だからこそ、Ace とは別プロダクトとして扱う。観測から気楽に始めて、組織 OS に育てる、という段階的導入が成立する。

> **現在の開発状況**
>
> - **EIS** —— OSS として公開済み（`brew install machuz/tap/eis`）
> - **OrbitLens Ace** —— **現在開発中**。観測解釈層を先行フェーズとして作っている
> - **OrbitLens Ideal** —— Ace の出荷後に着手する後続プロダクト。本章で描くのは Ideal の**到達目標としての組織 OS 像**であり、プロダクトはまだ存在しない
>
> 書籍で構造駆動論を固定し、プロダクトは段階的に配線する——この並走自体が、構造駆動を**書籍で閉じない**ための設計だ。

## 5. ブランドとしての OrbitLens

![OrbitLens](https://raw.githubusercontent.com/machuz/eis/main/docs/images/blog/sdo-ch9-orbitlens-inversion.svg)

- 外部表記：**OrbitLens**（本書・LP・外部ドキュメント）
- 内部表記：**orbitlens**（リポジトリ・コード・社内ドキュメント）
- 法人登記：**Orbitlens, Inc.**

三層プロダクト構成——それぞれ**否定で立ち位置を定義し、構造駆動ループの 3 位置**を担う：

- **OrbitLens Ace** —— 管理ツールではない。**組織の現在地と不足を読む**プロダクト（内部観測）。可視化・解釈・ギャップ発見
- **OrbitLens True** —— 採用プロダクトではない。**外から足りない構造を埋める**プロダクト（外向き）。構造ギャップを人材市場に翻訳し、候補を提案する
- **OrbitLens Ideal** —— 運用支援ツールではない。**今いる人員で組織を回す**組織 OS（内向き）。配置・介入・記録・再観測・文化化（開発着手前）

3 つは独立したプロダクトだが、**同じ構造語彙の上に立っている**——7 軸・アーキタイプ・3 層・変換・介入 3 層・文化シグナル。共有語彙が、プロダクト間を配管する。

**Ace で穴が見える → True で外から埋める／Ideal で中で動かす → Ace で再観測する**——3 つのプロダクトは構造駆動ループの 3 位置を担い、Ace を起点と終点にして閉じる。**提案の分離**が、この役割分担を明確にする：True の提案は**誰で埋めるか**（人材提案・外向き）、Ideal の提案は**どう動かすか**（運用提案・内向き）。

## 6. 組織論が SaaS になる瞬間

この本を書き始めたとき、SaaS は**ひとつの観測解釈ツール**として構想していた。CLI が吐く JSON を取り込み、可視化とアラートを返す、程度の設計だ。

書き進めるうちに、**観測解釈だけでは構造駆動論の半分しかプロダクト化できない**と気付いた。介入設計（第 6 章）、文化シグナル（第 7 章）、再観測ループ（第 8 章）——これらも**観測を前提にした配管**であり、**同じ語彙の上で動く**。観測だけを SaaS 化して、介入配管を手作業に戻すのは中途半端だ。

一方で、**観測の軽さと組織 OS の重さを 1 プロダクトに押し込むのは破綻する**。観測は 1 画面で朝イチに開けることに価値があり、組織 OS は運用合意と導入プロセスを伴う重さがある。両者を 1 プロダクトでやろうとすると、Ace の気楽さが消えるか、Ideal の深さが出ないか、どちらかに倒れる。

そこで、**2 つの SaaS に分ける**判断に至った：

- **Ace**（観測 SaaS）—— 軽く、手元に、気楽に
- **Ideal**（組織 OS SaaS）—— 重く、配管として、組織に

この 2 つの**組み合わせ**で、構造駆動エンジニアリング組織論の全体がプロダクトとして立ち上がる。Ace だけを使うチームも、Ace + Ideal で組織 OS として運用するチームも、**同じ語彙の上で段階的に進める**。

組織論は、**書籍**か**コンサル**で閉じるのが 100 年の慣習だった。書籍は理論を残せる。コンサルは一時的な運用を残せる。**SaaS だけが、観測と介入の配管を、組織に残せる**。

構造駆動は**観測を土台に設計されている**ので、配管ごと SaaS にできる。**これが、組織論がプロダクトになる歴史的瞬間**だ。

## 7. 境界線 — SaaS は何をしないか

Ace にも Ideal にも貫通する原則として、**やらないこと**を明示的に残す。

- **介入そのものは、現場の人間の仕事**。Ideal は介入の材料を出し、記録し、次の観測に繋ぐところまで。「人間が人間に声をかける」瞬間を SaaS が代替することはない
- **信頼の蓄積、非言語の了解、人間関係の機微**は観測しない（第 8 章の観測倫理に従う）
- **評価・懲戒への直接接続は構造上禁止**。Ace のダッシュボードも Ideal の介入記録も、**組織の自己修正のための材料**であり、個人の査定資料ではない。混ぜた瞬間、現場は自然な発話を止めて文化が死ぬ
- **代替的な観測パス**は意図的に複数残す。Slack、会議、1on1、レビュー——人間の目がアクセスできる経路を潰さない

観測 → 解釈 → 介入（第 6 章の 行動 / 出力 / 蓄積）→ 記録 → 再観測 の 5 段のうち、SaaS が担うのは **観測・解釈・記録・再観測** の 4 段（Ace が観測・解釈、Ideal が記録・再観測の配管）。**介入を人間が行う** という 1 段が、意図的に SaaS の外に残される。人間の判断を SaaS に預けないための境界線だ。

### 評価にしない設計 — 実装レベルの保証

第 8 章で「技術が先、リテラシーが後、は事故になる」と書いた。Ace の UI は、この原則を**実装レベルで保証する**方向で設計されている。評価の道具になりうる形を、慎重に避ける。

具体的には、組織アカウントに**3 段階の可視化モード**が設定される：

- **Observation（デフォルト）** —— 匿名。メンバーは Member A, B, C... として表示。構造パターンのみで、個人の合計スコアは出さない。心理的安全性を最大化する設定
- **Context** —— 名前と役割が見える。相対シグナルとトレンドは表示するが、**個人の合計スコアは依然として隠す**。関係性の理解のためのモード
- **Full Insight** —— 全プロフィール、個別スコア開示。運用判断のための設定で、組織として明示的な合意を伴って選ぶモード

デフォルトが最も匿名な **Observation** になっているのは意図的だ。**情報開示は段階的に、組織の合意を伴って**——これが Ace の姿勢。エンジニアから「勝手に個人評価に使われる」不安を構造的に遮断するため、ドメインモデル側はフル情報を保持したまま、**表示層でフィルタする**設計を取る。逆方向の誤り（後から匿名化を足そうとして漏れる）を、モデル起点で封じている。

さらに、**自分のシグナルをどこまで共有するかをエンジニア自身が選択できる**方向で設計を進めている。組織側が開示を一方的に強制する形ではなく、**エンジニアの選択を尊重する枠組み**として組み上げる。リテラシーが整う前にフル開示しない、リテラシーが整った先で段階的に開く——第 8 章で論じた社会的着地を、UI の構造制約として取り込むための設計だ。

## 8. 現場で何が変わるか

**Ace が入ると**：

1. **解釈が加速し、示唆が出やすくなる** —— 観測が「見る」から「読む」に変わる。情報が適切な粒度でブレイクダウンされ、レイヤー間・時期間での参照が即座にできる。ダッシュボードを開いた瞬間に「どのモジュールがどう動いているか」が頭に入る
2. **観測の運用負荷が消える** —— PR マージごとに EIS が再計算され、ダッシュボードが更新される。EM が手元で CSV を捏ねる世界は終わる
3. **組織の変化が時系列で残る** —— これまでの組織論は「現時点の組織の健全性」は語れても「3 ヶ月前から今への推移」は語れなかった。Ace は時間軸で変化を保持する

**Ideal が入ると（将来）**：

4. **介入が属人化から解放される** —— 1on1 のテンプレ、レビューの語彙辞書、組織変更のチェックリストが管理される。**辞めた人の頭の中にあった運用が、組織に残る**
5. **文化シグナルが可視化される** —— 第 7 章で定義した 3 バケツが統合ダッシュボードとして常駐する。文化の定着度が、数字で読める
6. **経営と現場が同じ言語で組織を議論できる** —— 取締役会でも全社会議でも、同じ構造シグナルと介入設計の 3 層で話せる

Ace から入り、Ideal で深める——この段階的導入が、構造駆動の実装パスになる。

## 9. 次章への接続

観測が手元に置かれ、組織 OS が配管され、構造駆動が段階的にプロダクトとして立ち上がる世界——そこで問われるのは「組織を管理するとは何か」ではなく、**「OS を作るとは何か」**だ。

次章（最終章）は、構造駆動エンジニアリング組織論が最終的に何を作っていたかの総括。**管理でも文化でもない、組織の OS を作る**、の意味を閉じる。
