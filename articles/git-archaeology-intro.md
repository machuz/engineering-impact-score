### はじめに

こんにちは。うっすら転職してポーカー系の事業でBEインフラリードをやっています。エンタメはいいぞ。

エンジニアとして働く中で、ずっとモヤモヤしていたことがある。**エンジニアの実力をどう定量的に評価するか**という問題だ。

コミット数？　行数？　PR数？　どれも一面的すぎる。コミットの粒度なんて人によって全然違うし、行数は自動生成やpackage-lock.jsonで簡単に水増しされる。PR数に至っては、typo修正1件もアーキテクチャ設計変更1件も同じ「1」としてカウントされる。

「なんとなくこの人は強い」「この人は政治はうまいけどコード書けない」——そういう直感はある。でも直感は属人的で、説明責任を果たせない。給与交渉の場で「なんとなく」は通用しない。自分は採用ミスったことがないのでこの辺の直感は外さないのだが、採用後に一緒に働く中でどれだけやったかを定量的に測れる何かがほしい。
そして、うちの今のチームが、自分の経歴上でもかなり強いのだけど、それが実感できる指標が欲しい。

きっかけは単純だった。

**このチームは強い。しかしそれを説明する言葉がなかった。**

「うちのチーム、実はめちゃくちゃ強いんですよ」——そう言いたかった。でも証拠がなかった。声の大きさでも、政治力でもなく、**コードに残った事実**でチームの強さを語りたかった。

酒を飲みながらClaude Codeと壁打ちした結果、**gitの履歴データだけで、体感と一致する戦闘力スコア**が作れたので共有する。名前は **Engineering Impact Score**——略して **EIS**（エイス、"ace" と読む）。

※ 異論は認める。が、少なくとも自分のチームでは驚くほど体感と一致した。

### 7つの軸で評価する

![EIS Framework Overview：Git履歴から7軸スコア・3軸トポロジー・Gravityを算出](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/engineering-impact-framework-diagram-fixed.png?v=5)

いきなり結論から。エンジニアの戦闘力を以下の7軸で測る。

| 軸 | 重み | 何を測るか |
|---|---|---|
| 生産量 (Production) | 15% | **1日あたりの変更行数**（絶対値。基準値 `production_daily_ref` で設定可能、デフォルト1000） |
| 初回品質 (Quality) | 10% | fix/修正コミット率の低さ。一発で正しいコードを書く力 |
| コード生存力 (Survival) | **25%** | 書いたコードが今も残っているか。**時間減衰付き**で計測。設計の耐久性 |
| 設計力 (Design) | 20% | アーキテクチャファイルへのコミット数。設計判断への関与度 |
| 領域幅 (Breadth) | 10% | 関与リポジトリ数。横断的に動ける力 |
| 負債清掃力 (Debt Cleanup) | 15% | 他人の技術負債をどれだけ片付けるか。チーム貢献度 |
| 不可欠度 (Indispensability) | 5% | 自分しか触れないモジュール数。影響度とリスクの両面 |

最初は5軸で作っていたが、実際に計測してみると「他人の負債を黙々と片付ける人」や「抜けると危険な属人化」を十分に表現できなかったため、負債清掃力と不可欠度を追加した。

**コード生存力の重みが最も大きい（25%）**のがポイント。後述するが、時間減衰付きの生存力はこのモデルの核心テーゼ。「今も強い設計を書き続けているか」を最も重視する。

品質（10%）の重みが低いのは、コミットメッセージベースのプロキシが粗いから。不可欠度（5%）が低いのは、「強いから必要」と「引き継がれていないから独占」のシグナルが混在するため。重みの設計にも意味がある。

そして7軸を**BE / FE / Infra で分離して計算する**。これをやらないとBEのコード量でFEランキングが汚染されたり、その逆が起きる。

### 各指標の計算方法

#### 生産量 — 「1日あたりの変更行数」で絶対値化

最初はコミット数で測っていたが、途中で行数（insertions + deletions）に切り替えた。さらに**絶対値化**した。

理由は2つ。**1コミットに100行変更する人と、1行変更を100回コミットする人が同じスコアになるのはおかしい**。そして相対値だと、**全員が怠惰な現場でも最も書いた人が100点になってしまう**。

絶対値化の方法：著者の最初のコミットから最後のコミットまでの期間で総変更行数を割り、**1日あたりの変更行数**を算出。これを設定可能な基準値（`production_daily_ref`、デフォルト1000）で割って0-100にスコア化する。

```
production_score = min(changes_per_day / production_daily_ref × 100, 100)
```

こうすることで、**1人しかいないドメインでも遅い人は遅いスコアが出る**し、**組織間でもある程度比較可能**になる。

ただし以下は除外する。でないと数値が壊れる：

- `package-lock.json`, `yarn.lock` — ライブラリ更新で万単位の行が動く
- `docs/swagger*` — 自動生成
- `mock_*`, `*.gen.*` — コード生成

```bash
❯ git log --all --no-merges --format="COMMIT:%an||%s" --numstat
```

#### 初回品質 — 「修正」という言葉が語るもの

```
quality    = 100 - fix_ratio
fix_ratio  = fix_commits / total_commits × 100
```

fix系コミットの判定はこう：

![fix判定コード](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/blog/ch1-code-fixdetect.svg)

conventional commitsを使っていないメンバーは日本語で「〜を修正」と書く。これを拾わないと実態と乖離する。

fix率が高いということは、多くの場合、**自分が書いたコードに修正が入りやすい**ということだ。初回品質の近似値としては十分使える。ただし、能動的な改善や大きな設計変更もfix系コミットに含まれうるので、設計力と合わせて解釈する必要がある。品質の重みを10%に抑えているのも、このプロキシの粗さを考慮しての設計。

#### コード生存力 — 時間減衰で「今の実力」を測る

ここがこの指標のキモ。

素朴なgit blameだと、「3年前に大量のコードを書いて、今は何もしていない人」が高スコアになる。これは正しくない。**今もアクティブに良い設計を書き続けているか**が知りたい。

そこで**指数関数的な時間減衰**をかける。

![時間減衰生存力の計算](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/blog/ch1-code-survival.svg)

| 経過日数 | 重み | 意味 |
|---|---|---|
| 7日 | 0.96 | ほぼ満額。最近書いたコード |
| 30日 | 0.85 | まだ高い |
| 90日 | 0.61 | 半分強 |
| 180日 | 0.37 | 約1/3。半年前 |
| 365日 | 0.13 | かなり低い。1年前 |
| 730日 | 0.02 | ほぼゼロ。2年前 |

![時間減衰の重みカーブ：730日間の指数関数的減衰](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/survival-decay-curve.png?v=5)

これにより、退職したメンバーのスコアが自然に下がり、**今のチームの実力**がより正確に反映される。創業期に大量のコードを書いたけど今はいない人が延々とランキング上位に居座る、という問題を解決できる。

素のblame行数（Raw Survival）は別途「コードベースの土台を誰が作ったか」の可視化として使う。戦闘力スコアに組み込むのは時間減衰付きの方。

#### Dormant vs Robust — 生存力の中身を分離する

コードが「生き残っている」と言っても、その意味は2つある。

- **Dormant Survival**（休眠生存）: 誰も触らないモジュールにコードが残っている。耐久性ではなく、単に放置されているだけ
- **Robust Survival**（変更圧下の生存）: **他の人が頻繁に変更しているファイルの中で**コードが生き残っている。活発に開発が続くモジュールで書き換えられずに残っているコードだけをカウントする

通常のSurvivalはこの2つを区別しない。EISはこれを分離して測る。

この区別が効いてくるのがStyle分類だ。全体のSurvivalは低いがRobust Survivalはそこそこある人は、試行錯誤しながらも変更圧に強いコードを生み出している（**Resilient**スタイル）。逆に、Survivalが高いのにRobust Survivalが低い人は、誰も触らないモジュールにコードが残っているだけの可能性がある（**Fragile**状態）。

#### 設計力 — 触るファイルでわかる

「アーキテクチャファイル」を定義して、そこへのコミット数を数える。

BEなら：

- `repository/*interface*` — リポジトリインターフェース
- `domainservice/` — ドメインサービス
- `partprocess/` — 共通ビジネスロジック
- `delegateprocess/` — 外部システム統合
- `router.go`, `middleware/` — APIルーティング
- `di/*.go` — 依存性注入

FEなら：

- `core/`, `stores/`, `types/`, `hooks/`, `lib/`, `ui/`

これらのファイルに頻繁にコミットしている人間は、**システムの骨格を作っている人間**だ。

正確には「設計力」というより「設計関与度の近似値」。直接的な設計の上手さを測っているわけではないが、設計に関与していない人間が良い設計をしている可能性は低い。近似として十分機能する。

**重要なのは、このパターン定義がプロジェクトごとに異なるということ。** デフォルトではクリーンアーキテクチャやDDD系のディレクトリ構成を想定しているが、プロジェクトの構造に合わせてカスタマイズしないと設計スコアの精度が出ない。たとえばProtobuf定義（`*.proto`）やマイグレーションファイル、Terraform定義がアーキテクチャの中核を担うプロジェクトなら、それらをパターンに含めるべきだ。

逆に言えば、**「どのファイルがアーキテクチャファイルか」を定義する行為自体が、チームの設計思想を言語化するプロセスになる**。「うちのプロジェクトで設計判断が現れるのはどこか？」という問いに答えることが、チームの設計基準を明確にする第一歩になる。

#### 領域幅 — 何リポ触れるか

全リポジトリのうち、何個にコミットがあるかを数えるだけ。単純だが、**「自分の領域」しか見ない人と、チーム全体を横断的に見る人**の差がきれいに出る。

#### 負債清掃力 — 誰が掃除しているか

fixコミットで変更された行の「変更前のblame」を調べる。つまり、**誰のコードを誰が修正したか**を追跡する。

![負債清掃の追跡](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/blog/ch1-code-debt.svg)

これを入れた瞬間、チーム内の「黙々と他人のバグを直している人」が可視化された。逆に「大量に書くけど周囲にfix作業を発生させている人」も浮き彫りになる。

**注意**: 負債の生成数+清掃数が合計10件未満のメンバーは、サンプルが少なすぎて比率が極端に振れるので参考値として扱う。

#### 不可欠度 — Bus Factorの裏返し

各モジュール（ドメインディレクトリ）のblame行分布を調べて、**80%以上を一人が占めているモジュール数**を数える。

![不可欠度の検出](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/blog/ch1-code-indispensability.svg)

不可欠度が高い = そのメンバーが抜けたら壊滅するモジュールがある。これは強みであると同時にリスク。重みを5%に抑えているのは、「強いから必要」なのか「誰も引き継がなかったから独占」なのか区別しづらいため。不可欠度は評価指標であると同時に、**引き継ぎ優先度を示すアラート**でもある。

### 正規化とスコア計算

指標には**絶対値スコア**と**相対値スコア**の2種類がある。

**絶対値**（組織間で比較可能）：

- 生産量: `min(1日あたり変更行数 / production_daily_ref × 100, 100)`
- 品質: `100 - fix率`（そのまま0-100）
- 負債清掃: 独自の0-100スケール

**相対値**（ドメイン内で正規化）：

- 生存力、設計力、領域幅、不可欠度: ドメイン内の最大値で正規化

**BE / FE / Infra / ファームウェアで分離して計算する**。これをやらないとBEのコード量でFEランキングが汚染されたり、その逆が起きる。ドメインは設定ファイルで明示指定するか、ファイル拡張子から自動検出される。

総合スコアは重み付き加算：

```
score =
  production       × 0.15
  + quality        × 0.10
  + survival       × 0.25
  + design         × 0.20
  + breadth        × 0.10
  + debt_cleanup   × 0.15
  + indispensability × 0.05
```

100点満点。

注意してほしいのは、**この指標はかなり辛口**だということ。100点満点と聞くと学校のテストを想像するかもしれないが、全然違う。7軸すべてで高スコアを出すのは構造的に難しい。生産量を上げれば品質が下がりやすいし、設計力を上げるにはアーキテクチャファイルに継続的に関与する必要がある。負債清掃力に至っては、自分が高品質なコードを書きつつ他人のバグも直さないと上がらない。

目安としてはこんな感じ：

| 総合スコア | 評価 | 適正時給感（税別） |
|---|---|---|
| 80〜100 | 代替不可能なコアメンバー。ここに到達する人はチームに1〜2人 | ¥12,000〜20,000 |
| 60〜79 | 準コア。強い | ¥9,000〜15,000 |
| **40〜59** | **シニアエンジニア相当。この指標で40を超えていれば普通に優秀** | **¥7,000〜11,000** |
| 30〜39 | ミドル | ¥6,000〜9,000 |
| 20〜29 | ジュニア〜成長中 | ¥5,000〜8,000 |
| 〜19 | ジュニア | ¥3,500〜6,000 |

**40でシニア**。これがこの指標の厳しさだ。7軸の相対評価なので、全軸でそこそこの数字を出すだけでも相当な実力が必要になる。「うちのチーム、シニアが40点しかないんだけど……」と思ったら、それは正常。40点台のエンジニアは市場でも十分に戦える人材だと思ってもらっていい。

**ここで大事な前提を1つ。** EISが測っているのは **「このコードベース上での貢献」** であって、エンジニアとしての絶対的な能力ではない。スコアが高いということは「このコードベースにおいて、その人のコードが残り、設計に影響を与え、負債を掃除している」ということ。逆に言えば、Survivalが高いのは設計が悪くてリファクタできないから消えずに残っているだけ、という可能性もある。スコアが直感と合わない場合、それは人の問題ではなくコードベースの設計品質の問題かもしれない。正しい設計のもとでどれだけスコアを保てるか——そこが本質的な実力になる。

![スコアガイド](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/score-guide.png?v=5)

### 指標の分布で見えるエンジニアの「型」——3軸トポロジーモデル（v0.9.0〜）

スコアを眺めていると、面白いことに気づく。**7つの軸の分布パターンで、そのエンジニアの「型」が浮かび上がる**。

v0.9.0で、従来の「単一アーキタイプ＋セカンダリ」方式を廃止し、**3軸トポロジーモデル**に刷新した。旧方式では「アーキテクトビルダー」「チャーンプロデューサー」のように1つのラベルに全てを詰め込んでいたが、実際のエンジニアはもっと多面的だ。「何を貢献するか」「どう貢献するか」「今どのフェーズにいるか」は独立した次元であり、それぞれを分離して記述するほうが現実に即している。

#### 3つの軸

| 軸 | 問い | 選択肢 |
|---|---|---|
| **Role**（何を貢献するか） | チーム内でどんな役割を果たしているか | Architect, Anchor, Cleaner, Producer, Specialist, — |
| **Style**（どう貢献するか） | どういうやり方で成果を出しているか | Builder, Resilient, Rescue, Churn, Mass, Balanced, Spread, — |
| **State**（ライフサイクルフェーズ） | 今どんな状態にあるか | Active, Growing, Former, Silent, Fragile, — |

出力は `Role / Style / State` の3つ組で表現される。たとえば：

- `Architect / Builder / Active` — 設計をリードし、自分で大量に作り、今もアクティブ
- `Cleaner / Balanced / Active` — 負債清掃が主な貢献、バランスの取れたスタイル、活動中
- `Architect / — / Former` — かつてアーキテクトだったが、今は活動していない
- `Producer / Churn / Active` — 大量に書くがチャーン（書いたそばから壊れる）スタイル
- `— / Spread / Silent` — 明確な役割なし、拡散的、実質的なアウトプットなし

「—」は該当する特徴が検出されなかったことを示す。無理にラベルを付けないのも設計の一部。

#### Role（何を貢献するか）

| Role | 説明 | 主なシグナル |
|---|---|---|
| **Architect** | システムの骨格を設計している | 設計↑、生存↑ or Robust生存↑ |
| **Anchor** | チームの品質基盤を支えている | 品質↑、生存↑、負債清掃○ |
| **Cleaner** | 他人の技術負債を片付けている | 負債清掃↑ |
| **Producer** | 大量のコードを生産している | 生産↑ |
| **Specialist** | 特定領域で圧倒的な成果 | 不可欠↑、幅✕ |
| **—** | 明確な役割が検出されない | — |

#### Style（どう貢献するか）

| Style | 説明 | 主なシグナル |
|---|---|---|
| **Builder** | 作って残す。設計し、実装し、それが生き続ける | 生産↑、生存↑、設計↑ |
| **Resilient** | 試行錯誤しながら変更圧に強いものを残す | 生産↑、生存↓、Robust生存○ |
| **Rescue** | レガシーコードを引き取って書き直す | 生産↑、生存↓、負債清掃↑ |
| **Churn** | 書いたそばから壊れる無限リワークループ | 生産○、品質✕、生存✕、生産-生存差≥30 |
| **Mass** | 大量に書くが残らない | 生産↑、生存↓ |
| **Balanced** | 突出はないが全軸でそこそこ | 全軸○ |
| **Spread** | 広く浅く。コードでの貢献は薄い | 幅↑、他✕ |
| **—** | 明確なスタイルが検出されない | — |

#### State（ライフサイクルフェーズ）

| State | 説明 | 主なシグナル |
|---|---|---|
| **Active** | 現在アクティブに貢献中 | 減衰付き生存力が高い |
| **Growing** | まだ量は出ていないが丁寧に成長中 | 生産↓、品質↑、fix率低 |
| **Former** | かつて貢献したが今は活動していない | Raw生存↑、減衰生存↓ |
| **Silent** | 席を占めているが耐久性のあるアウトプットがない | 生産↓、生存↓、負債清掃↓ |
| **Fragile** | コードは残っているが、それは誰も触っていないだけ | 生存↑、生産↓、品質<70 |
| **—** | 明確なフェーズが検出されない | — |

#### 各組み合わせの読み方

**Architect / Builder / Active** — 設計し、自分で作り、他人のコードも直す——フルパッケージ。チームの中核。このタイプが抜けるとプロダクトが止まる。品質スコアが低めに出ることがあるが、これは「攻めた設計変更を積極的にやっている結果としてfix率が上がっている」パターンが多い。

**Architect / Balanced / Active** — 設計力が高く、コードも耐久性があるが、必ずしも大量には書かない。設計をして委譲する古典的なアーキテクト。

**Architect / — / Former** — Raw生存力と減衰付き生存力のギャップで検出される。コードベースにはまだ大量にコードが残っているが、時間減衰をかけると急激に下がる。退職・異動後も「この人のコードが支配しているモジュール」が残っている状態。**ハンドオフ優先度アラート**として機能する。

**Producer / Churn / Active** — 大量にコードを書くが品質が極めて低く、ほとんどがfix/revertコミット。コードも残らない。**書いたそばから壊れる無限リワークループ**を生んでいる。

**Producer / Rescue / Active** — 生産量は高く生存力は低いが、負債清掃力が高い。**他者のレガシーコードを引き取って書き直している**パターン。コードが残らないのは品質の問題ではなく**古いコードを書き換えている**から。

**Producer / Resilient / Active** — 全体の生存力は低いが、変更圧の高いモジュールでのRobust Survivalはそこそこある。**試行錯誤を経て変更圧に強いものを生み出している**。苦悩しながら作って壊しをやりつつ、結果として変更圧に強いものを書くエンジニアは、正当に評価されるべきだ。

**Producer / Mass / Active** — たくさん書くが、残らない。負債清掃力も低い＝自分のバグを他人に直させている。一見「頑張っている」ように見えるので危険。**チームの負債製造機**になっている可能性がある。

**Anchor / Balanced / Active** — 生産量はトップではないが、fix率が低く、書いたコードがしっかり残る。さらに他人の負債も黙々と片付ける。地味だが、チームにとってはめちゃくちゃありがたい存在。

**— / Spread / Active** — 多くのリポに顔を出しているが、生産量も生存力も設計力も低い。**顔は広いがコードで貢献していない**。自分の経験上もこのタイプがワークしていない高単価人材であったことがある。幅だけ高くて総合20点台。数字は残酷なほど正直だ。

**— / — / Silent** — 生産量が低く、書いたコードも残らず、他人の負債も片付けない。**チームに席を占めているが、耐久性のあるアウトプットを何も生んでいない**。最も厄介なのは、目立った問題を起こさないので見落とされやすいこと。※コミット100以上の人にのみ適用。

**— / — / Fragile** — 生存力が高く生産量が低い、かつ品質が微妙（70未満）。**コードが生き残っているのは品質が高いからではなく、誰も触っていないから。**変更圧が高まった瞬間にこのコードは崩壊する。

**Specialist / Builder / Active** — 特定領域で圧倒的な成果を出すが横の広がりがない。属人化リスクはあるが、チームにとって貴重。

**— / — / Growing** — まだ量は出ていないがfix率が低く丁寧に書いている。生産量と設計力が伸びれば化ける。

![エンジニアトポロジー：生産量 vs 生存力の散布図](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/engineering-archetypes-paper-figure.png?v=5)

### エンジニア重力（Gravity）——構造的影響力の可視化（v0.10.0〜）

7軸スコアと3軸トポロジーで「この人はどんなエンジニアか」がわかるようになった。しかしもう一つ、チームの中での**構造的な影響力**——つまり「この人が抜けたらどれくらい構造が崩れるか」を表す指標が欲しかった。

v0.10.0で追加した **Gravity（重力）** がそれだ。

```
Gravity = Indispensability × 0.40 + Breadth × 0.30 + Design × 0.30
```

不可欠度（属人化の度合い）、領域幅（横断的な関与）、設計力（アーキテクチャへの関与）——この3つを組み合わせることで、**コードベースに対する構造的な引力**を近似する。Gravityが高い人間は、チームの設計判断・モジュール所有権・横断的な知識のハブになっている。

**ただし、Gravityは高ければ良いというものではない。**

重力には「良い重力」と「悪い重力」がある。品質が高く、生存力もあるエンジニアの高い重力は**健全な構造的影響力**——その人の設計が組織の骨格になっている状態。一方、品質が低く生存力も微妙なのにGravityだけ高い場合は**脆い構造的依存**——その人に依存しているが、コード自体は壊れやすい。

これを色で表現する：

```
health = Quality × 0.6 + RobustSurvival × 0.4

Gravity < 20  → dim gray  (low influence)
health ≥ 60   → green     (healthy gravity)
health ≥ 40   → yellow    (moderate)
health < 40   → red       (fragile gravity)
```

実データで見ると面白い。BE側のmachuはGravity 97で緑。品質57は高くないが、設計100・生存力100・不可欠度43と全方位で高く、構造的影響力が健全に分散している。一方、別のドメインであるメンバーはGravity 100で赤。不可欠度が極めて高く大半のモジュールを独占しているが、品質スコアが振るわない。**「この人が抜けたら崩壊する」かつ「このコード自体も壊れやすい」** ——最も危険な状態が、赤い重力として可視化される。

Gravityは総合スコアには組み込まない。総合スコアは「この人の戦闘力はいくつか」を答える指標であり、Gravityは「この人の構造的影響力はどうか」を答える別の次元の情報だからだ。ターミナル出力では各メンバーの行に色付きで表示される。

**コードから逆算してエンジニアの特性を読み取り、さらにそこからチームの構造を逆算する**——Gravityはその構造逆算の第一歩だ。

### なぜ売上KPIだけでは技術組織の強さが見えないのか

経営者やマネージャー向けに書く。

「売上が伸びているからエンジニア組織は問題ない」——これは危険な誤認だ。売上KPIは**プロダクトの市場適合性**を測っているのであって、**技術組織の健全性**を測っていない。

たとえ話をする。売上は「車の速度」、技術組織の状態は「エンジンの状態」だ。エンジンが壊れかけていても、下り坂なら速度は出る。速度が出ているからといって、エンジンが健全だとは限らない。

技術組織の健全性を知るには、**売上からは見えない4つの指標**を見る必要がある。

#### 1. コードの耐久性（Survival）

書いたコードが半年後も設計変更なしで動いているか。

生存力が低い組織は、**同じ機能を何度も書き直している**。売上には直接影響しないが、開発速度が徐々に落ちていく。3ヶ月ごとに「前に作った機能を作り直す」作業が発生するチームは、永遠にスケールしない。

#### 2. 技術負債の蓄積速度（Debt Rate）

新機能を追加するたびに、既存コードの修正が増えていないか。

負債清掃率が低い組織は、**新機能を1つ追加するたびに、既存のバグ修正が2つ発生する**ような状態になる。売上は伸びているのに「なぜか開発が遅い」と感じたら、これを疑うべき。

#### 3. バス係数リスク（Indispensability）

特定のエンジニアが抜けたら止まるモジュールがいくつあるか。

うちのチームでも、退職済みメンバーが97%のblame行を持つモジュールが存在する。その人が書いたコードの設計思想を理解しているメンバーが他にいないと、**バグ修正すら怖くてできない**状態になる。

#### 4. 設計判断の集中度（Design）

アーキテクチャの意思決定が誰に集中しているか。

1人のアーキテクトにすべてが集中している場合、その人の判断ミスがプロダクト全体に波及する。逆に、設計スコアが分散しているチームは、**設計レビューが機能している**証拠。

---

**売上が伸びていても、Survival低下 + Debt増加 + Bus Factor集中が同時進行していたら、スケーリング時に崩壊する。**

戦闘力スコアはこれらを定量化し、「投資すべきエンジニア」と「組織リスク」を同時に可視化する。エンジニアの評価に「なんとなく」しか使えない組織は、この数字を出すだけで景色が変わるはず。

### 既存の指標との違い

エンジニアの生産性指標は世の中にいくつも存在する。この戦闘力スコアがどこに位置するか整理する。

| フレームワーク | 何を測るか | 限界 |
|---|---|---|
| **DORA** | デプロイ速度と安定性 | チーム指標。コード品質や個人のインパクトは測れない |
| **SPACE** | 5次元の包括的評価（サーベイ+ツール） | サーベイ依存が重く、導入に3〜6ヶ月かかる |
| **McKinsey** | 組織の生産性 | [広く批判されている](https://newsletter.pragmaticengineer.com/p/measuring-developer-productivity)。アウトプット劇場を助長しうる |
| **行数 / コミット数** | 活動量 | 簡単にゲームできる。リファクタリングがペナルティになる |
| **コードチャーン** | 最近書いたコードの書き換え率 | リファクタリングと不安定性の区別がつかない |
| **バス係数** | 知識集中リスク | リスクの特定のみ。インパクトは測れない |
| **Git分析ツール**（Pluralsight Flow, LinearB等） | 活動量とサイクルタイム | コードが「いつ書かれたか」は測るが、「残ったか」は測らない |

**既存指標の共通の盲点は「その人が書いたコードが実際に生き残ったかどうか」を問わないこと。** DORAはコードが本番に届く速度を教えてくれる。この指標は、デプロイする価値があったかを教える。

時間減衰付き生存力はゲーミングにも強い。忙しそうに見せる作業ではスコアは上がらない——数ヶ月後にコードベースに残っているコードだけがカウントされる。負債清掃軸があることで、他人に仕事を発生させてスコアを上げることも構造的にできない。

### この指標の限界と読み方

完璧な指標など存在しない。これにも限界はある：

- **fix率はコミットメッセージ依存**。メッセージの書き方が統一されていないと精度が下がる。だから品質の重みは10%に抑えている
- **blameは大規模リポだと時間がかかる**。サンプリングが必要になる場合がある
- **FEはリファクタ頻度が高い**ので生存力が低くなりがち → だからBE/FEを分離する
- **行数はコピペでも増える** → 生存力と設計力で相殺される
- **負債清掃力はサンプル数に依存**。fixコミットが少ないメンバーは比率が極端に振れるので閾値（10件未満は参考値）を設けている
- **不可欠度は「強さ」と「組織の歪み」が混在**。だから重みを5%に抑えている

万能ではないが、**「直感の裏付け」としては十分すぎるほど機能する**。少なくとも「なんとなく」よりは100倍マシだ。

#### 設計が整った現場ほど精度が出る

この指標は**コードベースの設計品質が高いほど精度が上がる**性質を持っている。

クリーンアーキテクチャやDDDでレイヤーが明確に分離されている現場なら、「設計ファイルに触っている＝設計判断をしている」が成立するし、Survivalが高い＝変更圧に耐える良い設計だったと解釈できる。逆に、設計方針がなくファイル構成が混沌としている現場では、Survivalが高くても「誰も触れない死んだコード」かもしれないし、設計スコアが低くても「そもそもアーキテクチャファイルが明確に存在しない」だけかもしれない。

つまり、**この指標で精度が出ないこと自体が、設計の悪さのシグナル**になる。「計測してみたけど体感と全然合わない」と感じたら、指標がおかしいのではなく、コードベースの構造が計測に耐えられていない可能性がある。設計に投資すること自体が、チームの評価精度を上げるインフラになっている。

#### スコアは個人の能力だけを反映しない

重要な注意点がもう一つある。**スコアが低い原因がエンジニア個人にあるとは限らない**。

たとえば：

- **仕様が曖昧で頻繁に変わる現場**では、エンジニアがどれだけ優秀でもfix率は上がるし、書いたコードは仕様変更で書き直される。品質と生存力が下がるのはエンジニアのせいではなく、プランニングの問題
- **設計レビューや意思決定が遅い組織**では、エンジニアが次の実装に着手できず生産量が出ない。Productionが低いのはエンジニアが遅いのではなく、組織のボトルネック
- **要件定義者とエンジニアの認識がズレている現場**では、作ったものが「違う」と言われて作り直しになる。Survivalが低いのは設計力の問題ではなく、上流工程の問題

つまりこの指標は、**エンジニア個人の能力だけでなく、その人が置かれた環境——仕様の品質、プランニングの精度、意思決定のスピード——も反映している**。チーム全体のスコアが低いなら、それは個々のエンジニアを責める前に、組織のプロセスを疑うべきシグナルかもしれない。

逆に言えば、**プランニングと仕様の品質を上げれば、エンジニアのスコアは自然と上がる**。スコアはエンジニアの通信簿であると同時に、組織の健全性のバロメーターでもある。

そしてエンジニア側から見ると、このスコアを上げるには**言われたものをそのまま作るだけでは足りない**。仕様が筋の良いものかを自分で判断し、おかしければ議論に参加し、プロダクトを深く理解した上でコードに落とす必要がある。生存力の高いコードは、正しい仕様理解と正しい設計判断の上にしか生まれない。つまりこの指標は、**プロダクトを愛し、仕様から関わるエンジニア**を自然と高く評価する構造になっている。そうすればあなたのスコアは自然と伸びるだろう。それが本当の「戦闘力」なのかもしれない。

#### Survivalを読むときの注意点

素のgit blameには、初期に大量実装した人が有利になりやすいバイアスがある。誰も触れないコードでも、残っている限りその人の行としてカウントされるからだ。

本記事ではこの問題を避けるために、戦闘力スコアには素のblameではなく**時間減衰付きのSurvival**を使っている。2年前のコードの寄与は0.02まで下がり、「今も残る設計」を重視する形になっている。

それでもなお、Survivalは単独で解釈すべきではない。**DesignやBus Factorと合わせて見る**ことで、良い設計だから残っているのか、誰も触れないから残っているのかを見分ける必要がある。生存力トップと設計力トップが一致していれば健全。一致していなければ、コードベースに「触れない負債」が眠っている可能性がある。

#### あなたの会社のエンジニアの偉い人、戦闘力いくつですか

この記事を読んで「うちでもやってみよう」と思った人に問いたい。

**あなたの会社でエンジニアリングの意思決定をしている人の戦闘力は、チーム内で何位ですか？**

内製組織を本気で強くしたいなら、**技術的な意思決定者の戦闘力がチーム内で最も高い（少なくとも上位にいる）状態**が絶対に必要。なぜなら、設計判断の正しさはコードに現れるから。アーキテクチャを決める人間が自分でコードを書き、そのコードが残り続けている——この状態が、チームの設計品質を担保する唯一の方法だと思う。

逆に、戦闘力が低い人間が意思決定者のポジションにいると何が起きるか。**現場で実際にコードを書いている高戦闘力メンバーの設計判断が、戦闘力の低い上位者によって却下される**。これは組織にとって致命的。

**コードベースに痕跡が残っていない人間が設計判断をしている組織**は、構造的に歪んでいる可能性がある。もし計測してみて、あなたの上司の戦闘力がチーム内で下位だったら——それは指標のせいではなく、組織の構造的な問題かもしれない。

### 3ヶ月ごとに計測する

この指標は定点観測してこそ価値がある。

3ヶ月前と比べて生存力が上がっていれば、そのメンバーは**設計力が伸びている**ということ。fix率が下がっていれば、**初回品質が改善している**ということ。負債清掃率が上がっていれば、**チームへの貢献度が上がっている**ということ。

逆に、生産量だけ上がって他が横ばいなら、**量は増えたが質は変わっていない**ということ。

数字は嘘をつかない。

### CLIツール化した

この指標、最初はClaude Codeに計算させていた。14リポ・10名規模でトークン消費50万〜150万。定額プランじゃないと気軽に回せない。

**計算式が固まったので、CLIツールにした。**

```bash
❯ brew tap machuz/tap
❯ brew install eis
```

これだけ。AIトークンゼロ、API keyゼロ、クラウド依存ゼロ。

```bash
# Analyze current repo
❯ eis analyze .

# Auto-discover repos under a directory
❯ eis analyze --recursive ~/projects

# With config
❯ eis analyze --config eis.yaml --recursive ~/projects
```

`eis.yaml` で著者名のエイリアス（同一人物の名寄せ）、除外著者、ドメイン定義、アーキテクチャパターン、軸の重みなど全てカスタマイズできる。

```yaml
aliases:
  "John Smith": "john"
  "J. Smith": "john"
exclude_authors:
  - "dependabot[bot]"
domains:
  backend: ["api-*", "worker"]
  frontend: ["web-*", "app"]
  firmware: ["raden"]
exclude_repos: ["deprecated-service"]
production_daily_ref: 1000
tau: 180
```

実行すると、7軸スコア・総合点・3軸トポロジー（Role / Style / State）・Bus Factorリスクがターミナルに色分けで出力される。数秒〜数分で完了する。

![ターミナル出力例](https://raw.githubusercontent.com/machuz/engineering-impact-score/main/docs/images/terminal-output.svg?v=0.11.0)

**自分のチームの戦闘力、コマンド一発で測れます。**

### おわりに

エンジニアの評価は難しい。でも難しいからといって「定量化できない」と諦めるのは違うと思っている。

gitの履歴という、誰もが持っているデータだけで、ここまで体感と一致するスコアが作れた。もちろんこの指標だけで全てを判断するべきではない。コードレビューの質、ドキュメンテーション、チームへの貢献——数字に現れない価値はたくさんある。

でも、**まず数字で語れる部分は数字で語る**。その上で、数字に現れない部分を定性的に補う。この順番が大事だと思う。

定量評価のない組織で「あの人は優秀」「あの人は微妙」という空気だけが漂っている状態は、健全ではない。**評価には評価する側の解像度が必要だ**。その解像度を上げるための道具が、この戦闘力スコアだと思っている。

AIによる開発が増えても、良い基盤の上で、AIと筋の良い対話ができて、良いコードを書かせられているかがこの指標でわかるはず。AIとの対話で良い基盤を作れるかも重要。そこに人間の判断の余地はまだ残っている。汎用AIによってその余地も消えるかもしれないがw

そしてもう一つ。汎用AI時代が本格的に来たとき、**AIが書いたコードの「剛性」を測る指標**としてもこれは使えるかもしれない。AIが大量のコードを生成する時代に、そのコードの生存力はどうか。半年後にblameで残っているか。それとも人間が全部書き直しているか。負債清掃率で見たとき、AIは負債製造機なのかクリーナーなのか。——gitは人間もAIも等しく記録する。「AI導入で開発速度が上がった」という定性的な報告ではなく、**AIが書いたコードの生存力・負債率を定量的に追跡する**。それが次のフェーズの評価軸になる気がしている。

最後に一つだけ。**このスコアは人間の価値を還元するものではなく、コードベースに現れる技術影響力の近似値である**。コードレビューで後輩を育てる力、ドメイン知識の深さ、チームの心理的安全性を作る力——数字に現れない貢献は山ほどある。この指標はあくまで「gitに残る痕跡」から読み取れる部分を定量化しただけ。万能ではない。でも、ゼロよりは無限にマシだ。

世界最強のチームを作るために、まず現在地を正しく知ることから始めよう。あなたのチームの戦闘力、計測してみてほしい。思っている以上に、数字は正直だから。
